Mission Control "Operation" card: the middle card that answers "what happened, and can it ship?" — a derived verdict banner, a one-sentence plain-language mission line, and a curated, read-only proof summary. The verdict is derived, never decorative: `ALL CLEAR` requires affirmative evidence (a fully green suite and passing review and security gates), `GATE HOLD` requires a real failing security gate, and with missing/partial run data the banner shows an honest neutral state ("No run data yet" / "In progress" / "Not fully verified") — never a false ALL CLEAR over an unknown run. Because the server does not yet emit review/security gates, a real green run honestly reads neutral today and will light ALL CLEAR once those gates are wired. The proof summary is a short (≤8), curated list of proof lines built from real per-run data (FR-01.47) — it is emphatically NOT the terminal: no xterm, no pty, no WebSocket, no input (guarded by test), and the real embedded terminal stays byte-identical in the Files & Terminal tab. Durations always render `n/a` (never synthesized), an empty log yields an honest empty summary (never invented lines), and untrusted event-log text is sanitised (control/bidi characters stripped, length-capped) before render. Every sentence comes from the Mission narrator (FR-01.54); a design gate routes to A14's surface with an honest placeholder until it lands. State is consumed from A11's `useMissionState` (no second derivation). Only the `/tasks/:taskId` Mission-tab (done + live) visual baselines move (FR-01.56).
