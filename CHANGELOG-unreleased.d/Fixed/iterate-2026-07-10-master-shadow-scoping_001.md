Single-session master-shadow identity is now project-scoped and guard-visible (campaign webui-deep-audit D07, F06/F34). When two projects were registered from a duplicated directory — which copies `shipwright_run_config.json`, and therefore the `runId`, verbatim — clicking Launch on the second project's single-session pipeline card could reuse or resume the FIRST project's master conversation, because the client's master-shadow lookup matched only `parentRunMaster` + `runId` across the whole task list. The lookup is now scoped to the launching project's id, so each project launches its own master. On the server, a direct-API `POST /tasks/:id/launch { masterRun: true }` on a plain task now stamps `parentRunMaster` + `runId` (server-derived from the re-read run-config) on the launched task, so every attached master is visible to the double-master guard that returns `409 master_run_already_attached` — previously an API-launched master left those fields unset and a second master could attach to the same run undetected.
