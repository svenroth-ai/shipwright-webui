The Windows production server scripts (start-server-production.ps1 and stop-server.ps1) now honor a custom server port set via the PORT environment variable, matching their macOS/Linux (.sh) counterparts (campaign webui-deep-audit D14, F35). Previously they always targeted port 3847: an operator who ran the Command Center on a different port had the old server left running on the custom port while the "stop"/restart reported success, hit an "address already in use" error on the new server, and got a misleading "did NOT come up on port 3847" message. Both scripts now read PORT once (defaulting to 3847 when unset or invalid) and use it consistently for stopping the old server, launching the new one, checking it came up, and every status message. This is a Windows operator-script fix only; nothing in the running Command Center changes.
