Two kinds of request failures now return the right, actionable status code instead of a generic "Internal server error" (campaign webui-deep-audit D17, F29 + F33). First: when another window or process is briefly holding the lock on the app's task file, several actions — closing, forking, creating, launching, or deleting a task, dismissing an inbox item, saving settings, or deleting a project — used to fail with a blunt 500 error, even though the same situation on the task-edit action already returned a friendly "locked, please retry" (409) response. All of these now return that same retryable "locked, retry" response, so the app can simply try again instead of showing a hard error. Second: sending an empty or malformed request body to the settings-save or project create/edit actions used to crash into a 500; it now returns a clear "invalid request body" (400) response. Nothing changes for normal, well-formed requests — the fix only replaces misleading 500s with the correct 409/400 the app already knew how to handle.
