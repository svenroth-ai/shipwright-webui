A corrupt or half-written projects list file no longer bricks startup (campaign webui-deep-audit D08, F07). `ProjectManager.load()` previously ran a bare `JSON.parse` with zero corrupt-file tolerance while its writers are non-atomic fire-and-forget saves — so if a save was interrupted mid-write (a force-kill via the documented `taskkill` dev workflow, a power loss, or a `tsx watch` restart) the truncated `projects.json` made the server exit FATAL on every subsequent boot with an unattributed parse error. Load now mirrors the `sdk-sessions-store` recovery pattern: an empty/whitespace file loads as an empty registry, and a truncated/unparseable file is quarantined aside to `projects.json.corrupt-<timestamp>-<uuid>` (best-effort; never re-throws) while the app continues with an empty project list — so the Command Center always starts. Saves are now written atomically (stage to a `.tmp-<uuid>` file, then rename into place) so a force-kill mid-save lands on the throwaway temp file and can never truncate the live registry. The recovery + atomic-write helpers live in the new `server/src/core/project-registry-io.ts`.
