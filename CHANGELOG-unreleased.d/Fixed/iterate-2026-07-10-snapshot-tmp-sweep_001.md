Interrupted terminal-snapshot writes no longer leave secret-bearing staging files on disk forever (campaign webui-deep-audit D19, F26). When the app saves a terminal's on-screen state it first writes a temporary staging file and then renames it into place. If the app was shut down (or a Windows file lock persisted) at exactly the wrong moment, that staging file — which holds the full terminal screen, possibly including secrets — was orphaned, and nothing ever cleaned it up: the delete-a-task cleanup only removed the finished snapshot, and the periodic disk sweep only matched scrollback logs. Now a task deletion also reclaims that task's orphaned staging files, and a boot-time plus once-a-day sweep removes any staging file older than an hour (old enough to be provably abandoned, so a save that's still in progress is never touched). Live snapshots and scrollback logs are untouched. Nothing visible changes for current tasks; the fix only stops the slow leak of abandoned terminal-state files.
