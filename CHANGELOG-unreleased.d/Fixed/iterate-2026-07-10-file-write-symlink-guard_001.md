A project file that is secretly a shortcut (symlink) with a `.md` name can no longer be used to overwrite a non-markdown file through the in-app markdown editor (campaign webui-deep-audit D09, F09). `PUT /api/external/projects/:id/file` validated the markdown-only allowlist on the request path — the symlink's *name* — while the write followed the link to its real target, and the existing realpath check only confirmed the target stayed inside the project. So a within-root symlink such as `notes.md -> shipwright_run_config.json` (or `-> package.json`, or a build script) slipped past both checks and the save landed on the non-markdown file — defeating the markdown-only write boundary and the read-only run-config invariant, up to code-execution via a build-script / git-hook target. The write path now `lstat`s the target without following it and refuses any final-component symlink outright with a distinct `403 symlink_forbidden`, and the atomic save now replaces the requested directory entry itself (never dereferencing a symlink swapped in after the check), so a `.md`-named symlink can no longer redirect a write to its target. Regular markdown edits, the `ETag`/`If-Match` optimistic-concurrency flow, and GET/read behavior are unchanged.
