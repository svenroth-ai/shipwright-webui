# Conventions ‚Äî shipwright-webui

## Linter / Formatter

- **Linter**: oxlint ‚Äî `npm run lint` (`oxlint .`) wired in both `client/` and `server/`; CI runs it per workspace (iterate-2026-05-19-oxlint-and-cors-env)
- **Formatter**: _none detected_
- **TypeScript strict**: no
- **.editorconfig**: _none_

## Project-specific rules

TypeScript runs in strict mode across both halves; `npm run build` (which runs `tsc`) exits 0 on both halves. The historical "4 baseline errors tracked" carve-out from ADR-035 was retired by ADR-080 ‚Äî server now ships verbatim mirrors of `Task`, `GlobalSettings`, `Project` under `server/src/types/`; cross-package imports back into `client/src/types/` are rejected by the comment-aware drift-guard at `server/src/test/no-cross-package-imports.test.ts`. Hono routes live in `server/src/routes/` (one file per resource); the external-launch surface is registered from `server/src/external/routes.ts` and split into 9 sub-routers under `server/src/external/{tasks,launch,transcript,inbox,actions,preview,tree,file,run-config}/` (Campaign C / C2, 2026-05-26). React components in `client/src/components/` are grouped by UI area (`external/`, `terminal/`, `triage/`, `settings/`, `sidebar/`, `wizard/`, `common/`); large area-roots have been split into per-area subfolders (`external/BubbleTranscript/`, `external/NewIssueModal/`, `external/TaskDetailHeader/`, `external/SmartViewer/`). Files stay under 300 lines ‚Äî anything larger gets split, with the bloat-baseline + pre-commit hook (see [Pre-commit hooks](../../CLAUDE.md#pre-commit-hooks)) as the enforcement floor; documented deep-module exceptions (ADR-101 `pty-manager.ts`, ADR-103 `terminal/routes.ts`) ride on explicit ADRs.

Data fetching uses TanStack React Query with sequential 1 s polling for transcript updates (no SSE, no chokidar). Styling is TailwindCSS 4; accessible primitives come from Radix UI. Markdown rendering is bespoke via `react-markdown` + `remark-gfm` + `rehype-highlight` + `strip-ansi` ‚Äî `@assistant-ui/*` packages are explicitly forbidden (DO-NOT guard #4 in CLAUDE.md). Auto-scroll is CSS-first: `overflow-anchor: auto` + `scroll-padding` on the scroll container, with a small ref-based `useAutoScroll` as the safety net for Chrome+polling.

Commits follow Conventional Commits (`feat:`, `fix:`, `refactor:`, `test:`, `docs:`, `chore:`). The repo has no root `package.json` ‚Äî `server/` and `client/` are independent workspaces, each with its own `npm run dev | build | test | lint | typecheck`. CI runs through `.github/workflows/ci.yml`; there is no detected linter / formatter / editorconfig at the root level (each workspace owns its own). E2E lives in `client/e2e/` (Playwright); the multi-service detector pivots Playwright setup into `client/` because that is where `package.json` and `playwright.config.ts` live.

Three categories of regression guards are spelled out in CLAUDE.md and must be respected: (1) the WebUI never writes into Claude's JSONL or `shipwright_run_config.json`; (2) the chat composer must not be re-introduced (Spec 35 is the regression fence); (3) phase strings (`build`, `plan`, `design`, `iterate`) and slash-command names must not be hardcoded ‚Äî they come from `/api/external/projects/:id/actions`. A meta-test (`client/src/test/doc-sync.test.ts`) keeps the file-map in CLAUDE.md honest.

## Commit messages

- Use Conventional Commits: `feat:`, `fix:`, `refactor:`, `docs:`, `test:`, `chore:`
- Scopes should reflect module boundaries (e.g., `feat(auth): ...`)

## Files

- Keep files under 300 lines; split larger modules.

## Architecture rules (post-Plan-D'' variant a + Iterate 2; embedded-terminal auto-execute via ADR-068-A1)

1. **Webui never spawns Claude.** `core/launcher.ts` produces command strings; the embedded-terminal pane auto-executes them via a client-side WS data-frame after an explicit Launch / Resume / Relaunch click (ADR-068-A1), or the user copies and runs them in their own terminal. The pty-manager shell-only whitelist (ADR-067) is the architectural enforcement line. Regression guard: Playwright spec `35-no-chat-panel.spec.ts`.
2. **Task state is derived from `~/.claude/projects/<encoded-cwd>/<uuid>.jsonl` + the persistent store.** Session UUID is pre-bound at task creation via `crypto.randomUUID()`.
3. **Discovery is filename-first.** `<uuid>.jsonl` is the primary match (PoC finding 1); first-line sessionId is a secondary sanity check.
4. **Transcript endpoint is stateless.** `GET /api/external/tasks/:id/transcript?fromByte=<n>&expectFingerprint=<fp>` ‚Äî multi-tab support comes for free.
5. **UTF-8-safe chunking.** Server reads are cut on `\n` boundaries only.
6. **Torn-read retry budget.** `core/session-watcher.ts` retries EBUSY/EPERM/EACCES/ENOENT up to 6 attempts with 50‚Üí1600 ms backoff.
7. **No SSE for transcript.** Sequential 1 s polling on the client via `useTaskTranscript`.
8. **No chokidar.** Heartbeat-free; watcher state is derived on demand from mtime probes.
9. **Plugin dirs must be re-passed on every launch.** `--plugin-dir` does not reliably survive `--resume`.
10. **MIN_SUPPORTED_CLI is pinned.** See `core/cli-compat.ts`. Anything older shows a banner via `/api/diagnostics`.

### Preview-capability precedence (Iterate 3, plan ¬ß 2.1)

Iterate 3 introduces a Preview dev-server spawn path (not Claude ‚Äî see ADR-044). Three sources interact:

1. **Profile `stack.frontend` presence** ‚Äî the capability gate (is this project a frontend project at all?).
2. **Profile `dev_server.command`** ‚Äî the spawn target (which command actually starts the dev-server).
3. **`.shipwright-webui/actions.json` ‚Üí `actions.preview.enabled`** ‚Äî the policy override (user-level opt-out).

`stack.frontend` AND `dev_server.command` must both be present for `<PreviewButton>` to render. `actions.preview.enabled = false` hides it regardless. Boot-time coherence check warns when `stack.frontend` is set but `dev_server.command` is missing (button would render, spawn would 500). The full diagram lives in [`architecture.md`](architecture.md).

## DO-NOT regression guards (Iterate 2 / ADR-035)

1. **DO NOT write into Claude's JSONL files under `~/.claude/projects/`.** Webui is a read-only polling observer of that directory. Title sync uses Claude's first-party `--name` CLI flag at launch time, not JSONL mutation or a sidecar file.
2. **Auto-scroll pattern is CSS-first** (carve-out: virtualized branch ‚Äî see ADR-063). `overflow-anchor: auto` + `scroll-padding` on the scroll container is the primary path. `useAutoScroll` (ResizeObserver-light, ref-based, ~50 LOC) is the safety net for Chrome+polling cases. Do NOT reach for stale libraries like `react-scroll-to-bottom`. See ADR-035. **Virtualized-branch carve-out (ADR-063):** in `BubbleTranscript.tsx` the scroll container's inline style is `overflow-anchor: showVirtualized ? "none" : "auto"`. Browser scroll-anchoring fights TanStack-Virtual's DOM recycling on scroll-up ‚Äî the browser anchors on a row the virtualizer has just unmounted, then frame-by-frame fights `translateY` corrections ‚Üí flicker. In the virtualized branch only `useAutoScroll` (ref-based, survives DOM rotation) drives auto-scroll. Do NOT re-set `overflow-anchor: auto` unconditionally there.
3. **DO NOT re-introduce a chat composer.** External-launch architecture is load-bearing. See ADR-034. Spec 35 fails the build if a chat-* surface re-appears.
4. **DO NOT re-add `@assistant-ui/*` packages.** Rendering is bespoke via `react-markdown` + `remark-gfm` + `rehype-highlight` + `strip-ansi`. The full stack is in `client/package.json` already.
5. **DO NOT run `claude --resume <uuid>` as a webui-initiated side-effect command** while the user's session may be active in their terminal. SQLite lock + JSONL interleave risk (EBUSY on Windows). Title updates take effect on next user-initiated launch only.
6. **Multi-writer state files** (`sdk-sessions.json`, any future sidecars) MUST use `proper-lockfile`, not just temp-file + rename. The PATCH /tasks/:id endpoint surfaces ELOCKED as 409 so the client can retry instead of silently overwriting.
7. **Type-system isolation between workspaces.** Server and client are independent npm workspaces (no root `package.json`); `server/tsconfig.json` enforces `rootDir: ./src`, so cross-package imports of the form `from "../../../client/..."` break `tsc`. Shared shapes (`Task`, `GlobalSettings`, `Project`) live in `server/src/types/` as verbatim mirrors of `client/src/types/*.ts`; each header names its canonical origin. Drift is caught by two complementary tests: `server/src/types/action-schema-sync.test.ts` (content-parity for `action-schema.ts` fields) and `server/src/test/no-cross-package-imports.test.ts` (comment-aware regex sweep rejecting `from "../../../client/..."`, `import("../../../client/...")`, and any-depth relative re-exports). The prior "4 baseline errors tracked, no-regression" policy (ADR-035 ¬ß Consequences) is RETIRED ‚Äî `npm run build` exits 0. Do NOT add cross-package imports as a shortcut. See ADR-080.
8. (Iterate 3 ‚Äî ADR-044) **Schema v2 is write-on-touch.** `sdk-sessions.json` v1 rows load as `projectId: "unassigned"` in memory and rewrite to v2 on next mutation. DO NOT batch-rewrite on boot ‚Äî ADR-038 explicitly rejected that path. The `schemaVersion` header field must load both v1 and v2 during the forward-compat window.
9. (Iterate 3 ‚Äî ADR-044) **Preview spawn uses `shell: false`.** User-controlled `dev_server.command` strings + `shell: true` would be command-injection. All Preview subprocess entrypoints flow through `core/preview-session-manager.ts`, which tokenizes the command and spawns with `shell: false`. DO NOT add a parallel spawn path.
10. (Iterate 3 ‚Äî ADR-044) **Path-guard is `realpath + path.relative`, NOT `startsWith`.** Symlinks, unicode, and Windows junction points defeat prefix checks. All tree + file routes share `core/path-guard.ts`; null-byte input is hard-rejected.
11. (Iterate 3 ‚Äî ADR-044) **DO NOT hardcode `shipwright-run` / `shipwright-iterate` / phase strings (`build`, `plan`, `design`, ‚Ä¶) in components** ‚Äî read from `/api/external/projects/:id/actions`. Violations re-introduce the configurability debt iterate 3 paid down and will cause custom `.shipwright-webui/actions.json` installations to break silently. Meta-test `client/src/test/doc-sync.test.ts` keeps this file-map honest; grep your component for `shipwright-` literals before committing.
12. (iterate/multi-session-run-orchestrator-v2) **DO NOT write into the user's `shipwright_run_config.json`.** WebUI is a read-only observer of run-config. The framework's orchestrator owns every mutation; the only webui write surface remains `sdk-sessions.json` + `.shipwright-webui/actions.json` stub creation (existing). All run-config consumption flows through `core/run-config-reader.ts` (server) and `useRunConfig()` (client).
13. (iterate/multi-session-run-orchestrator-v2) **Phase-task launches must use the pre-bound `sessionUuid` from the run-config ‚Äî never re-generate.** The `/launch` route's `phaseTaskRef` branch re-reads run-config server-side and rejects mismatched session-uuids (`409 phase_task_session_uuid_mismatch`). The client never sends `sessionUuid` / `slashCommand` directly; only `phaseTaskRef.phaseTaskId`. Also: `phaseTaskRef` and `actionId` are mutually exclusive (`400 mixed_launch_intents`).
14. (iterate/multi-session-run-orchestrator-v2) **All pipeline-continuation entry points share `useContinuePipeline()`.** Master TaskCard CTA + Continue Pipeline modal + future TaskDetail header all funnel through the single hook so re-fetch + idempotent shadow-task lookup + verified launch happen as one atomic unit. Adding a parallel launch path bypasses the staleness re-check and re-introduces the race the architecture closes.
15. (iterate/multi-session-run-orchestrator-v2) **Schema v3 is additive + write-on-touch.** `sdk-sessions.json` gains optional `phaseTaskId` / `runId` / `parentRunMaster` fields. Loader accepts v1 + v2 + v3; persist always writes v3. Same forward-compat window as ADR-038. DO NOT batch-rewrite on boot.
16. (iterate/multi-session-run-orchestrator-v2) **Stale `in_progress` detection uses run-config timestamps only.** `phase_task.startedAt` / `claimAttemptedAt` against `config.updated_at` ‚Äî never JSONL mtime. WebUI must not depend on observing JSONL files for tasks it doesn't own a shadow of.
17. (Iterate 4 ‚Äî ADR-067) **pty-manager spawn target MUST be a whitelisted shell binary** (`pwsh / powershell.exe / cmd.exe / bash / zsh / sh / fish`); never `claude`. Plan-D'' compliance lives in the whitelist + the WS-upgrade Origin gate (loopback-only, mirrors existing CORS). `paste-image` + `append-gitignore` MUST flow through `realPathGuard` on the resolved target. 8 MiB image cap + 9 MiB Content-Length precheck + magic-byte mime sniff are non-negotiable. Pty.write of image-paste paths goes through `quotePathForShell(absPath, shellKind)`. WS upgrade is the AUTHORITATIVE pty creation path; `POST /spawn` is an idempotent prewarm. ADR-034 wording is amended: "Webui spawns no Claude process directly. Webui MAY host a neutral shell pane; Claude execution within that pane stays user-initiated (Strg+V + Enter)."

18. (ADR-068-A1) **Scrollback path-guard is `realpath` AT EVERY OPERATION** (not just boot-time). UUID `/^[0-9a-fA-F-]{36}$/` validated on every public `ScrollbackStore` method. File-naming: `<taskId>.log` (NOT sessionUuid ‚Äî multi-launch + resume + fork share one task). Rotate/read/clear/closeStream go through per-task `PQueue`; append uses WriteStream's own serialization. Rotation 4-state FSM (NORMAL ‚Üí ROTATING ‚Üí ROTATION_FLUSH ‚Üí NORMAL); overflow during rotation throws (cap 4 MiB). Replay-on-attach uses `pty.pause/resume` ‚Äî NEVER drop chunks (ANSI/UTF-8 corruption). POST `/close` keeps scrollback; `/clear-scrollback` is the destructive path.

19. (ADR-068-A1) **Auto-execute is via CLIENT-SIDE WS data-frame, NOT server-side `pty.write`.** `LaunchCoordinatorContext.dispatchAutoLaunch(commands, resume)` queues a `pendingLaunch` token; `EmbeddedTerminal` consumes it via `consumedTokens` dedup and sends `socket.send({type:"data", ‚Ä¶})` once `role === "writer"` AND prompt-readiness handshake clears (250 ms quiesce, 3 s cap). ## Learnings

- **jsdom in client/ does NOT implement `Blob.prototype.text()`.** Use a `FileReader` polyfill (`reader.readAsText(file)`) for any code path that needs to read a `File` object as a string in tests. See `client/src/components/settings/ActionsConfigCard.tsx:readFileAsText`. Captured during iterate-20260430-actions-upload-ui after `userEvent.upload(...)` + `await file.text()` failed with `TypeError: file.text is not a function`.
- **WebUI write surfaces** (current 2026-05-01): `~/.shipwright-webui/{projects,sdk-sessions,settings}.json`, `<project.path>/.shipwright-webui/actions.json` (stub create + upload + reset), `<project.path>/.shipwright-webui/<slug>.code-workspace` (POST `/api/projects` only, idempotent ‚Äî ADR-059). Every write inside a project path MUST go through `realPathGuard` to defeat symlink escape ‚Äî `/actions-stub` (legacy) and `/actions-upload` (FR-01.27) do; the workspace bootstrap is currently inside the user-supplied `body.path` directly so adding `realPathGuard` there is a small follow-up worth tracking (the existing `mkdirSync` block has the same shape). Add new write surfaces with the same guard or refuse to add them.
- **Iterate Phase Matrix "always" rows are not negotiable.** During iterate-2026-05-01-vscode-workspace I skipped the Spec Update step ("always" for FEATURE) and the architecture.md Write-Surfaces note, arguing "additive side-effect on existing FR-01.24 = no FR drift". User correctly pushed back. The shortcut is wrong even when the change is small: FR-01.24's acceptance criteria + the architecture.md write-surfaces paragraph were both load-bearing for compliance + future iterate drift checks, and skipping them quietly poisons the next run. Rule: when an iterate adds a new file output, a new endpoint behavior, or a new write surface ‚Äî even via a one-line server addition ‚Äî read the FR row that owns the touched route and update both the FR description AND the acceptance criteria, then update the architecture.md Data-Flow paragraph if a write surface or read surface changed. "It's just additive" is not a reason to skip; it's a reason the update is small.
- **Browser-coordinated layout heuristics are hostile to virtualization.** ADR-035's CSS-first auto-scroll (`overflow-anchor: auto`) was correct for the non-virtualized BubbleTranscript but produces a frame-by-frame flicker when virtualization engages: the browser picks an anchor row, the virtualizer unmounts it during scroll-up, and both then fight over the scroll position. ADR-062's stable-keys + animation-frame fix addressed real sub-bugs but couldn't stop the browser-side anchoring; ADR-063 was needed to switch off `overflow-anchor` in the virtualized branch only. Generalisation: any browser-coordinated layout heuristic that assumes stable DOM identity (`overflow-anchor`, `scroll-into-view` with `nearest`, position-anchoring of native dialogs, `:has()`-driven layout) becomes a footgun the moment virtualization recycles rows. Audit such heuristics before flipping virtualization on, and write the regression test as an inline-style assertion (jsdom-friendly) ‚Äî visual flicker itself is not testable without a real browser. **2026-05-01 update:** ADR-063 (which acted on this rule) backfired and was reverted same-day ‚Äî turning off `overflow-anchor` in the virtualized branch removed useful compensation when the browser-anchored row survived the overscan window, making overall scroll worse. The actual dominant flicker source for tool-heavy sessions was a different layer entirely (per-poll setState cascade ‚Äî see ADR-064). The generalisation about "browser layout heuristics + virtualization = caution" still holds; the specific fix in ADR-063 didn't.
- **Side-effects can be load-bearing ‚Äî measure before optimising perf.** Three iterates in a row (ADR-063, mermaid-hypothesis, ADR-064) tried to fix residual scroll-up flicker on long virtualized BubbleTranscript by reasoning from code-reading. All three made things worse and were reverted. The deeper finding: useTaskTranscript's per-poll `setResult({...})` cascade was *inadvertently* a "scheduled re-measure" for the TanStack virtualizer ‚Äî each poll re-rendered all visible BubbleRows, which re-ran `ref={virtualizer.measureElement}`, which caught up height drift from late-mounted markdown / rehype-highlight / font-swap. The visible flicker was the catch-up. ADR-064 killed the cascade and uncovered the underlying drift as content-jumping during scroll. Same shape as ADR-063: removing something that *looked* redundant exposed a load-bearing compensation. **Future debugging rule: when a perf optimisation candidate has wider implications (involves the React render loop, ResizeObserver, layout/scroll), the ONLY safe entry point is measurement ‚Äî DevTools Performance recording, instrumented render counters, layout-shift counts. Do NOT iterate on hypothesis-from-code-reading; the rendering stack is too interlocked.** **2026-05-02 confirmation (ADR-066):** the rule paid off. Phase 2 measurement (window.__instr2 over 30 wheel ticks) cleanly identified estimate-vs-measure as the dominant slow-scroll cascade source (mean +105 px positive delta from FALLBACK_ROW_PX=96, max +1743 px). The fix (localStorage cache + cold-cache warmup) was data-validated by Playwright two-pass probe (totalSize stable from first snapshot in both passes; +1197 px rehydration delta on cold-vs-warm comparison) BEFORE shipping. Fifth attempt at this bug, first time the measurement-first protocol was followed end-to-end; landed without a revert.

- **React 19 unmount cleanup is unreliable across hard navigation.** During ADR-066 the cache-persist effect's React unmount cleanup did NOT fire when Playwright navigated away via `page.goto`. The pagehide event fired correctly. Lesson: for any "save state on tab/page exit" use case, prefer the `pagehide` event listener (or `beforeunload` if you accept the dialog UX cost) over React's `useEffect` cleanup. React cleanup is a best-effort SPA-route fallback, not a reliable hard-navigation hook. Belt-and-braces: combine pagehide + a periodic `setInterval` flush so tab-crash / power-loss scenarios still capture recent state.

- **localStorage persistence + browser sandboxing.** Browser localStorage is per-origin and per-browser-profile. Playwright contexts have their own localStorage; the user's Chrome localStorage is sandboxed from anything the agent or its tools can run. To clear cached state in a real browser tab the user must paste a one-liner in DevTools Console (e.g. `Object.keys(localStorage).filter(k => k.startsWith('webui.virtualizerCache.')).forEach(k => localStorage.removeItem(k))`). Don't propose the agent run `taskkill` on the user's browser process ‚Äî kill scope is per-PID and risks unrelated work.

- **Radix Tabs needs userEvent.click in jsdom ‚Äî fireEvent.click silently doesn't trigger onValueChange.** Radix UI uses pointer events under the hood (pointerdown/pointerup) for its trigger interactions, and `fireEvent.click` from `@testing-library/react` only dispatches a click event. The result is a fast-fail symptom that's hard to diagnose: the click "happens" but state never changes, and the test waits for `data-state="active"` until timeout. `userEvent.click()` from `@testing-library/user-event` simulates the full pointer dance and Radix updates. Captured during iterate-2026-05-03-embedded-terminal-launcher Phase 3 ‚Äî first 4 of 7 tests timed out before the switch.

- **Radix `<Tabs.Content>` defaults to UNMOUNT-on-inactive ‚Äî use `forceMount` + CSS hide for any tab whose subtree owns load-bearing state.** Critical when xterm.js + a WebSocket live inside one tab: without `forceMount`, switching tabs tears down the Terminal instance + closes the socket + drops scrollback. Pattern: `<Tabs.Content value="x" forceMount className="data-[state=inactive]:hidden">`. External review (ADR-067 F3) caught this before build; the regression fence is a mount-counter spy in TaskDetailPage tests that asserts mountCount stays constant across toggles.

- **xterm.js v5 needs the scoped `@xterm/*` packages AND its CSS bundle.** `import "@xterm/xterm/css/xterm.css"` is mandatory ‚Äî without it the cursor doesn't render, `fit()` reports broken sizing, and the canvas stays opaque. The legacy `xterm` (unscoped) package is still on npm but unmaintained for new Node versions. `node-pty-prebuilt-multiarch` (the historical recommendation) lacks Node 22 / win32-x64 prebuilt binaries; switch to `@lydell/node-pty` (api-compatible, prebuilt for Win/Mac/Linux x64+arm64). Documented during ADR-067 build phase 1.

- **Hono multipart via c.req.parseBody() works for ‚â§10 MiB but pre-check Content-Length before buffering.** Pattern from FR-01.27 actions-upload, re-applied by FR-01.29 paste-image: 1 MiB headroom over the application cap (e.g. 8 MiB blob ‚Üí 9 MiB Content-Length precheck) gives a deterministic 413 fast-fail before the multipart parser allocates. Without the precheck the parser eats memory up to the runtime default (varies across @hono/node-server versions).

- **Playwright route-mocking: `apiFetch`-backed endpoints need a `{ data: T }` envelope; bare-`fetch` endpoints don't.** `client/src/lib/api.ts` `apiFetch()` returns `json.data`, so a mock must wrap its payload: `page.route('**/api/projects', r => r.fulfill({ json: { data: [...] } }))`. A bare array silently yields `undefined` downstream and the page renders its empty-state with no error. The triage endpoints (`/api/triage/*`) use bare `fetch` and consume the raw object (`{ items }` / `{ counts, total }`) ‚Äî no envelope. Captured during iterate-2026-05-15-triage-card-styling when a mocked `/api/projects` returned a bare array and the triage page showed "No projects registered" while the bare-fetch counts badge populated correctly.

- **xterm.js Ctrl+V uses the async Clipboard API, NOT DOM ClipboardEvent.** xterm's keybinding for Ctrl+V calls `navigator.clipboard.readText()` directly and writes the result via `term.paste()` ‚Äî it never fires a `paste` event on the focused textarea. Document/capture-phase `paste` listeners therefore never see real Ctrl+V (only right-click ‚Üí Paste menu, programmatic paste, or other DOM-event paths). To intercept Ctrl+V and route image clipboards through a custom upload, install `term.attachCustomKeyEventHandler` (synchronous return-value control: `false` suppresses xterm's default; you call `ev.preventDefault()` + drive `navigator.clipboard.read()` for the structured payload). Captured during iterate-2026-05-07 v0.8.3 AC-1 after v0.8.2's synthetic-event Spec 79 AC-3 test passed but real-browser Ctrl+V never reached our handler ‚Äî the v0.8.2 fix was a false positive, locked in only by the wrong test shape. Follow-up: every keyboard-shortcut AC needs a Playwright test that uses `page.keyboard.press()` against a real browser, not `dispatchEvent` on a fake textarea. **2026-05-08 (v0.8.5 AC-2 coda):** the v0.8.3 attachCustomKeyEventHandler implementation was REVERTED. The technical rule above (xterm's Ctrl+V uses async Clipboard API) is still correct and worth knowing for any future iterate that wants to intercept xterm keys. We simply chose not to act on it for image-paste because Alt+V via Claude Code's TUI clipboard pipeline (lands under `~/.claude/image-cache/`) is the supported path and produced reliable round-trips in the user's daily flow, while the v0.8.3 Ctrl+V interceptor never did. Surface size > value when the alternative path Just Works.

- **Image-paste path-of-record is split between WebUI and Claude Code's TUI.** WebUI handles shell-prompt pastes (`<task.cwd>/.shipwright-webui/pastes/`); Claude Code CLI handles TUI-session pastes (`~/.claude/image-cache/<sessionId>/`) via its own internal clipboard pipeline, bypasses our DOM listeners entirely. The split is intentional and surfaced openly in the in-product disclosure footer + `.shipwright/agent_docs/known_issues.md` ¬ß "Image-paste path-of-record". Do NOT attempt to migrate Claude's cache to `<task.cwd>/.shipwright-webui/pastes/` ‚Äî out of WebUI's surface, fragile reverse-engineering of Claude Code CLI internals, deferred indefinitely.

- **xterm `term.buffer.active` is wiped by ConPTY's `\x1b[2J\x1b[H` startup AFTER `replay_end` fires.** When the WS upgrade re-attaches and a fresh pty spawns, its initial output (which includes the screen-clear sequence on Windows ConPTY) arrives as live `data` envelopes RIGHT AFTER replay_end. Those bytes wipe the replay-painted scrollback content from the visible buffer ‚Äî meaning a buffer-scan AT replay_end captures the right state, but a buffer-scan even 100ms LATER returns 0. **Generalisation:** any "count tokens in xterm buffer" assertion must either (a) accumulate from the replay payload string BEFORE writing to xterm (the v0.8.7 AC-4 `replayBufferRef` pattern), or (b) snapshot the buffer DURING the replay_end callback (synchronous, before the next event-loop tick). Captured during iterate v0.8.7 AC-4 ‚Äî original spec said "scan term.buffer.active after replay_end"; empirical run against live stack returned 0 markers despite disk having ‚â•1; switched to a payload-string accumulator with per-chunk gating between replay_start/replay_end. Fallback buffer-scan stays for unit tests where the mock has no live wipe; production uses the accumulator value via `Math.max(accumulator, bufferScan)`.

- **`TERM=dumb` on the embedded-terminal pty silently disables ConPTY's startup screen-clear, so replay-on-attach must clear the xterm active area on the CLIENT.** `server/src/terminal/routes.ts createNodePtySpawnFn` sets `TERM=dumb` (escape hatch in chalk's `supports-color` for the brand-color hack). Side-effect: PowerShell + ConPTY don't emit `\x1b[2J\x1b[H` on shell start, so a freshly-spawned shell after replay-on-attach renders RELATIVE to the cursor's current position ‚Äî which after replay sits at row N+1 of the replay content. Result: replay sits in the active viewport, live shell appends below, Claude's ink TUI mis-positions because it expects a fresh terminal screen. Fix (v0.8.9): `EmbeddedTerminal.tsx onReplayEnd` writes `term.rows √ó \r\n` then `\x1b[H` to push replay into scrollback above + home the cursor. Generalisation: any "live process renders to xterm AFTER the WebUI has rendered something else" scenario must explicitly reset cursor / clear active area on the client side because TERM=dumb means the live process won't do it. Empirical fence: `term.buffer.active.baseY` increases when the pushdown succeeds (markerRow < baseY proves marker landed in scrollback, not active viewport).

- **External code review catches bug shapes that are easy to miss when the developer wrote the regex / control-flow themselves.** v0.8.7 ran `external_review.py --mode code` against the iterate diff and surfaced TWO HIGH-priority bugs the developer did NOT catch:
  1. `collapseSpan(span)` returned `pre + marker + LAST burst + post-last` ‚Äî silently DROPPING content between the first and last matched bursts. User output emitted between pty respawns would be lost on replay. (OpenAI severity: HIGH.)
  2. The accumulator's `chunk.slice(-overlap)` tail-retention under-counted markers when frames arrived smaller than the marker length (e.g. 1-byte chunks). (Gemini severity: HIGH.)

  Both bugs would have shipped without the review. Lesson: at medium+ complexity, run external code review BEFORE finalization ‚Äî it's faster than waiting for the first user-report and catches off-by-one / subtle-control-flow issues that the original author internalized as correct. The reviewer needs the iterate spec for context (what should it do?) plus the diff (what does it do?) ‚Äî both are mandatory inputs to `external_review.py --mode code`.

- **Drift-guard tests need WHOLE-CONTENT scanning + path-segment-bounded regex.** ADR-080's first iteration of `no-cross-package-imports.test.ts` did line-by-line regex scanning with `(?:\.\.\/)+client\/`. External review (OpenAI medium + Gemini medium) flagged two real gaps: (1) multi-line imports (`import {‚Ä¶}\n  from "../../../client/x.js"`) bypass per-line scans because neither line alone matches; (2) substring matching catches false positives like `from "../../foo-client-lib/baz"`. Fix: scan the whole comment-stripped content with `RegExp.exec` + map `match.index` back to a 1-based line via newline counting; use path-segment boundaries `(?:[^'"\/]+\/)*client\/` so `client` must be its own directory segment. Also covers dynamic `import("...")`. Generalisation: any "no X imports allowed" guard test should (a) work on stripped content, (b) span newlines, (c) anchor on path segments, and (d) ship sanity sub-tests proving each invariant.

- **`uv run` on shared-shipwright Python tools needs `--with openai`.** The shared scripts (`external_review.py`, ‚Ä¶) declare no inline `# /// script` deps and the uv environment is ephemeral per-invocation. `openai` is the SDK used to talk to OpenRouter (OpenAI-compatible API) ‚Äî invoking `uv run external_review.py` fails with "openai package not installed" until invoked as `uv run --with openai external_review.py`. Document this for the next iterate; future shipwright-shared cleanup may inline the deps.

- **`surface_verification.py` on Windows: pass `npm.cmd` (and other extension-bearing CLIs) explicitly.** F0.5 runs commands via `subprocess.run(cmd_list, shell=False)`. Windows `subprocess` does not auto-resolve `.cmd` / `.bat` / `.ps1` extensions on PATH ‚Äî `npm` returns `WinError 2 The system cannot find the file specified`. Use `npm.cmd --prefix <subdir> run <script>` as the runner string. Same applies to any CLI shim that ships as a Windows batch wrapper (yarn, pnpm, npx). Captured during ADR-080 F0.5; cli-surface evidence at `.shipwright/runs/iterate-20260509-tsc-baseline-fix/surface_verification.json`.

- **`0.0.0.0` is a bind-wildcard, NOT a routable HTTP destination.** When proxying or constructing a client-side URL, never use `http://0.0.0.0:port/` ‚Äî some HTTP clients fail outright, others behave inconsistently, and no real RFC says it should resolve. Use `http://127.0.0.1:port/` even when the server is bound to `0.0.0.0` (loopback always works against a wildcard listener on the same host). Captured during ADR-081 external code review (OpenAI HIGH finding) ‚Äî `client/src/lib/resolveProxyTarget.ts` falls through to loopback for the `open` profile.

- **Vite + Hono dev stack: when Hono moves off loopback, the Vite proxy target MUST follow.** `vite.config.ts` historically hardcoded `http://localhost:${PORT}` as the `/api` proxy target. The moment Hono binds to a non-loopback interface (`HONO_HOST=<ip>` or `SHIPWRIGHT_NETWORK_PROFILE=tailscale`), the proxy ECONNREFUSEs because Hono is no longer listening on loopback. ADR-081 introduced `client/src/lib/resolveProxyTarget.ts` that mirrors the Hono-bind decision. **Generalisation:** any time you change a server's bind interface, audit the consumers (proxies, healthchecks, smoke tests, scripts that curl localhost) for matching assumptions. External iterate review caught this gap pre-build (Gemini HIGH).

- **Tailscale CLI subprocess: timeout + CRLF + IPv6 noise are real edge cases.** `tailscale ip -4` can hang if the daemon is unresponsive (set `timeout: 2000` in `execSync` opts). On Windows, stdout has `\r\n` line endings (split with `/\r?\n/`). The output is per-host; multi-line output from a host with multiple Tailscale interfaces ARE valid IPv4s, but stdout MAY also include IPv6 lines or noise ‚Äî parse the FIRST valid IPv4 line (use `node:net.isIPv4` for strict octet-bound validation, never a hand-rolled regex which permits `999.x.x.x`). Captured during ADR-081.

- **`@types/node` for tsconfig.node.json only ‚Äî keep DOM-only browser code clean.** When client-side code (e.g. a Vite-config helper in `client/src/lib/`) needs Node APIs (`node:net`, `node:child_process`, etc.), do NOT add `@types/node` to the main `tsconfig.json` ‚Äî that would let arbitrary browser code import Node APIs and break the bundle silently. Instead: (1) exclude the Node-only source files from `tsconfig.json` `include`, (2) include them in `tsconfig.node.json` with `"types": ["node"]`. The client now uses this split: `resolve{Tailscale,NetworkProfile,Proxy}*.ts` are Node-only, never bundled into the browser. Captured during ADR-081 build phase.

- **`shlex.split(... posix=False)` preserves quotes in tokens on Windows ‚Äî pass relative paths to `surface_verification.py --runner`.** When the runner string contains quoted absolute paths like `--prefix "C:\..\server"`, Windows shlex keeps the quotes as part of the token; npm then treats the whole `"C:\..\server"` (including quotes) as a relative directory and emits ENOENT. Pass relative paths instead (e.g. `--prefix server`) and let `subprocess.run(cwd=project_root)` resolve them. Same applies to any cross-platform runner authored in shipwright that flows through F0.5. Also: `surface_verification.py`'s stdout parser doesn't recognize vitest's "Tests N passed" line shape ‚Äî pass `--tests-run N` explicitly so the gate doesn't fail-closed on `tests_run == 0`. Captured during iterate-20260514-lead-foundation-task-schema F0.5 (two failed attempts before the fix landed).

- **External LLM review pre-build catches narrow-write-surface footguns.** During iterate-20260514-lead-foundation-task-schema, OpenAI's review (MED-4) flagged that exposing daemon-owned ExternalTask fields (`claimToken`, `claimedBy`, `leadHandoff`, ‚Ä¶) through `store.create()` would let any POST `/api/external/tasks` body smuggle them in. The fix narrowed `create()` to accept only the 5 user-creatable fields; the route's `readLeadCreateFields` helper filters anything else. Without the pre-build review this would have shipped silently ‚Äî `Object.assign` style spreads are the natural way to plumb new fields through, and an attacker only needs to know the field names to bypass the daemon claim helper. Rule: at medium+ complexity, run `external_review.py --mode iterate` BEFORE writing tests ‚Äî narrow-write-surface findings reshape the test list itself.

- **A config-default flip in one iterate can silently falsify a signal another iterate depends on.** Iterate L (April 2026) established `altScreenActive` as the Claude-foreground proxy and gated the Resume CTA on `altScreenActive !== true`. Iterate J / ADR-098 (2026-05-13) then restored `CLAUDE_CODE_NO_FLICKER=1` as default-on ‚Äî which makes Claude render in the MAIN buffer, not alt-screen. Result: `altScreenActive` stays `false` during active Claude streaming, so Iterate L's gate always passed and the Resume button showed the whole time Claude was working. Nobody noticed because Iterate L's UAT loop ran while `NO_FLICKER` was still opt-in. Iterate M (2026-05-15) replaced the single signal with the compound `isPtyForegroundActive(task)` (`liveSession && firstJsonlObservedAt && now ‚àí lastPtyDataAt < 15_000`), which works in BOTH buffer modes. **Rule: when an iterate flips a config default, grep for every consumer of the signals that default influences ‚Äî a gate written against the old default is a dormant bug, not a stable contract. `altScreenActive` is buffer-mode-specific; `lastPtyDataAt` is buffer-mode-agnostic and should be preferred for any "is something running in the pty" question.** Captured during Iterate M (Sven UAT 2026-05-14 "Resume kommt zu schnell").

- **Synthetic stress may not reproduce a renderer-level bug ‚Äî visual UAT history is the load-bearing evidence.** Iterate K's xterm.js#5847 atlas-corruption workaround (v1‚Äìv10) was driven entirely by chronological user-UAT feedback ("Verschmierungen", "stale cursor", "alt-screen flicker"). The post-v8 empirical-validation attempt built two A/B probes (`probe-iterate-k-smearing-{ab,video}.mjs`) with a `?atlasMaintenance=off` kill switch and drove a synthetic pwsh `[Console]::Write` 256-color colorblast ‚Äî but could NOT visually reproduce the smearing with the workaround OFF. The bug needs Claude's specific cell-update churn pattern that naive per-cell color emit doesn't replicate. **Rule: a probe that validates control flow + pixel-diff plumbing is necessary but NOT sufficient for a "the bug is fixed" claim on a renderer-level artifact. When synthetic reproduction fails, say so explicitly in the ADR (ADR-099 ¬ß "Empirical-validation-attempt block" does this) and keep the probe scripts as regression infrastructure rather than presenting them as positive proof.** Captured during Iterate K.

- **`draft` is now a transcript-poll-sticky state ‚Äî and "sticky" is NOT a blanket skip.** iterate-2026-05-17-move-to-backlog made a task movable In-Progress ‚Üí `draft` ("Move to Backlog"). The `GET /transcript` state machine must therefore NOT transition a backlogged `draft` task back out. Two guard points in `external/routes.ts`: the `missing`-branch `‚Üíjsonl_missing` arm (gated `&& task.state !== "draft"`), and the `ok`-branch first-JSONL `‚Üíactive` arm. The `ok`-branch guard is the subtle one: a `draft` task CAN still legitimately bootstrap to `active` on its first JSONL ‚Äî that is how a freshly-created, never-launched task goes live (its launch produced a JSONL). The distinguishing signal is **`launchedAt`**: a *backlogged* `draft` task came from an In-Progress state, all of which are reached via `/launch` (which sets `launchedAt`); a *fresh* `draft` has no `launchedAt`. So the `ok`-branch suppresses `‚Üíactive` only when `task.state === "draft" && task.launchedAt`. **Rule: when adding any new transcript-poll transition, ask "does this fire for a `draft` task, and if so is it a fresh draft (bootstrap OK) or a backlogged draft (must stay sticky)?" ‚Äî the first version of the guard suppressed the bootstrap unconditionally and broke two `routes.test.ts` first-JSONL tests; external code review (gemini) caught the missing `ok`-branch case before that.** Captured during iterate-2026-05-17-move-to-backlog.

- **An iterate F0.5 web E2E must run against an ISOLATED server, not the user's live stack.** `server/src/config.ts` builds `registryDir` as `path.join(os.homedir(), ".shipwright-webui")` with **no env override** ‚Äî so a naively-started E2E server reads/writes the user's real `~/.shipwright-webui/sdk-sessions.json`, and every `request.post("/api/external/tasks")` in the spec pollutes the production task board. Fix: start the E2E server with `USERPROFILE` (and `HOME`) pointed at a throwaway temp dir ‚Äî `os.homedir()` on Windows resolves `USERPROFILE`, so `registryDir` relocates into the temp dir and the run is fully isolated. Combine with `SHIPWRIGHT_NETWORK_PROFILE=local` so the server binds `127.0.0.1:3847` while the user's production server keeps its Tailscale-IP `:3847` listener ‚Äî distinct bind addresses, no conflict (the `feedback_dev_vs_autostart_port_conflict` Option 4). Run the prod build (`node dist/index.js`), not `tsx watch`, for E2E stability. Captured during iterate-2026-05-18-edit-task-dialog F0.5.
- **A missing `.js` extension in a server ESM import surfaces as a confusing DOWNSTREAM type error, not an obvious import error.** During iterate-2026-05-18 `server/src/core/task-editability.ts` imported `ExternalTask` from `"./sdk-sessions-store"` (no `.js`). Under `moduleResolution: node16` tsc emits TS2835 for the import ‚Äî but it ALSO makes the imported type resolve to an error type, so `Pick<ExternalTask, "launchedAt">` then reported `launchedAt` as *required* (a `Pick` over an unknown/error type loses optionality), producing a baffling TS2345 at the call site. Rule: when a `Pick<>`-of-an-imported-type assignability error looks impossible, check the import line for a missing `.js` extension first ‚Äî fix the import and the phantom downstream error vanishes.

- **Terminal text paste must go through `term.paste()`, never a raw `socket.send`.** xterm's `term.paste(data)` normalizes line endings (`\r?\n` ‚Üí `\r`) AND wraps the content in bracketed-paste markers (`\x1b[200~`/`\x1b[201~`) when the focused app enabled `?2004h`. A raw `socket.send({type:"data",payload:rawText})` skips both ‚Äî so an app with bracketed-paste mode (Claude Code's TUI, PSReadLine) reads every `\n` as Enter and submits a multi-line prompt on its first line ("the paste got cut off"). The DOM `paste` listener carried this bug from ADR-067 until iterate-2026-05-18 routed its text branch through `term.paste()`. Rule: any path that injects clipboard / pasted text into the embedded terminal uses `term.paste()`; `socket.send({type:"data"})` is only for genuine keystroke / `onData` forwarding. Captured during iterate-2026-05-18-terminal-copy-paste.
- **`navigator.clipboard` is gated to secure contexts ‚Äî `execCommand('copy')` is not.** Over the Tailscale IP the WebUI is plain `http://` on a non-loopback host = NOT a secure context, so `navigator.clipboard` is `undefined` (both read and write). `lib/clipboard.copyText` already falls back to a temp-textarea + `document.execCommand('copy')`, which DOES work in a non-secure context ‚Äî so terminal *copy* works over Tailscale. *Paste* cannot: `navigator.clipboard.readText()` is unavailable and `document.execCommand('paste')` is browser-blocked for web content; the only non-secure paste path is the browser's native right-click ‚Üí Paste (a privileged UI action that fires a real `paste` ClipboardEvent). The real fix for keyboard paste over Tailscale is serving the WebUI over HTTPS. Captured during iterate-2026-05-18-terminal-copy-paste.

- **E2E against the prod build needs in-app navigation + a real task cwd.** Two traps surfaced wiring the F0.5 web run for iterate-2026-05-18-inbox-terminal-prompts. (1) The production Hono server (`node dist/index.js` serving `client/dist`) has NO SPA fallback ‚Äî `page.goto("/inbox")` 404s (only the Vite dev server rewrites unknown routes to `index.html`). Load `page.goto("/")` then click the in-app sidebar nav link. (2) A task seeded via `POST /api/external/tasks` with a non-existent `cwd` makes the terminal WS upgrade fail with `task_cwd_unresolvable` (500) ‚Äî no pty spawns, the terminal never reaches `ready`, and anything gated on terminal-ready (focus, replay) silently never happens. Seed with a real directory (`os.homedir()`).

- **Tailwind v4's preflight zeroes `list-style` ‚Äî rendered-markdown lists must restore it explicitly.** Tailwind v4 preflight resets `ol, ul { list-style: none }` globally. The shared `.markdown-body` CSS (`client/src/index.css`) set `padding-left: 2em` on lists but never `list-style-type`, so every rendered-markdown bullet/number marker was silently invisible everywhere `<MarkdownText>` renders ‚Äî Inbox `text_question` cards, `BubbleTranscript`, `SmartViewer`. Fix: explicit `.markdown-body ul { list-style-type: disc }` / `ol { decimal }` / nested `circle`/`square`; the `.markdown-body ul` selector out-specifies the bare `ul` preflight reset. Rule: any class meant to keep native list markers under Tailwind v4 must declare `list-style-type` explicitly ‚Äî preflight's reset is global and silent. Captured during iterate-2026-05-19-inbox-markdown-render.

- **A module-level `const` that reads `process.env` is baked at import time ‚Äî scrub the env in `vi.hoisted()`, never `beforeEach`.** `server/src/index.ts` resolves its CORS Trusted-Origin policy into a top-level `const corsOriginPolicy` (~L82) the instant the module is imported; `server/src/index.test.ts` does `import { app } from "./index.js"`. ESM `import` statements are hoisted above all other top-level code, so a `vi.stubEnv` / env mutation in `beforeEach` runs far too late ‚Äî the policy is already fixed. Only a `vi.hoisted()` callback runs before the imports resolve. Rule: when a test imports a module whose top-level code reads an ambient global (`process.env`, `Date.now`, ‚Ä¶), stub/scrub that global inside `vi.hoisted()` before the import. A dev shell exporting `SHIPWRIGHT_NETWORK_PROFILE=tailscale` was silently failing the default-loopback CORS test (1/1156) until this fix. Captured during iterate-2026-05-19-oxlint-and-cors-env.

- **The embedded-terminal writer slot had no liveness detection ‚Äî only WS `close`/`error` freed it.** A connection that died UNCLEANLY (OS sleep, browser crash, Tailscale half-open TCP ‚Äî the prod server binds the Tailscale IP) never fired that event, so the writer slot stayed pinned to a dead conn and every new tab showed a false "Read-only ‚Äî another tab is the active writer" banner. Two non-obvious traps surfaced while fixing it (iterate-2026-05-31-terminal-readonly-keepalive): (1) the `bufferedAmount` writer-stuck watchdog (ADR-069) is effectively **inert in production** because the manager's `conn` identity is the synthetic `connToken` `{taskId,t}`, which has no `bufferedAmount` ‚Äî so `connCapability` stamps every conn "missing" and the watchdog never evicts. It only ever evicts on backpressure anyway, never on liveness. (2) `readyState` stays `OPEN` on a half-open TCP until the OS timeout (minutes), so it can't drive fast reaping. The fix is a WS-native **ping/pong heartbeat** (`server/src/terminal/ws-heartbeat.ts`): the browser auto-pongs at the protocol layer (no client change), and on consecutive missed pongs the server `terminate()`s the socket, letting the EXISTING `onClose ‚Üí detach ‚Üí reader-promotion ‚Üí "writer-promoted"` chain clear read-only on the surviving tab WITHOUT a reload. Tolerate ONE transient miss (reap after 2) so an OS-sleep resume ‚Äî where the server interval and the peer wake on different ticks ‚Äî doesn't spuriously reap a healthy tab. Bloat-aware shape: all three terminal modules (`pty-manager.ts`/`routes.ts`/`ws-upgrade-handler.ts`) are at-ceiling ADR-101/103 exceptions where any net line ratchets and the pre-commit hook hard-blocks, so the logic went into a NEW neutral module (the ADR-103-sanctioned pattern, cf. `terminal-reset.ts`) with a single `startWsHeartbeat(ws)` wire into `onOpen` and a same-commit baseline bump (the anti-ratchet rule is `measured > current` against the on-disk baseline, so bumping `current` to the new LOC in the same commit passes ‚Äî no `--force`).

- **ADR-136** (2026-05-27): actionId-aware phase pill resolution; new-iterate never derives from title

- **A campaign's lifecycle `status` is NOT a reliable "is it done?" signal ‚Äî use `done>=total`.** The producer-owned top-level `status` only auto-flips `active`‚Üí`complete` inside the autonomous loop's `campaign_progress.py update-status` call (campaign-mode.md step 3g/4). A campaign driven via individual sub-iterate PRs (the common case for stacked campaigns) never gets that call, so it stays `active` forever even at 4/4. The Campaigns lane filter `selectActiveCampaigns` consequently rendered a 100%-done campaign card indefinitely (reported 2026-06-05, closing `compliance-detective-realign`). Fix (iterate-2026-06-05): `isCampaignDone(c) = status==='complete' || (total>0 && done>=total)`, checked BEFORE the `active` short-circuit. Rule: for any "is this campaign finished?" UI/logic, treat `done>=total` (with `total>0`) as the authoritative completion signal and `status==='complete'` as a secondary fast-path ‚Äî never `status==='active'` alone as "still running". The producer lifecycle is advisory, not load-bearing, in webui (we are read-only on it).

- **A cross-language wire contract is only "single source of truth" if the fixture is GENERATED BY the canonical binary ‚Äî and a boundary probe on fresh data catches what fixtures can't.** iterate-2026-06-10-triage-pending-delivery-badge gated the TS `pendingDelivery` projection against a fixture produced by subprocessing the REAL `triage_cli.py list --json` (not a re-implementation). The committed ASCII fixtures passed everywhere; the live round-trip probe with unicode titles then crashed the CLI itself (`UnicodeEncodeError`: `ensure_ascii=False` JSON written to Windows-cp1252 stdout) ‚Äî a real upstream bug no fixture would ever have caught, because regen would have crashed too and been "worked around". Rules: (1) parity fixtures must come from executing the canonical implementation (`regen-triage-fixtures.py` gate 3); (2) any `subprocess` of a Python CLI whose output contract is UTF-8 JSON needs `PYTHONUTF8=1` on Windows until the producer reconfigures its own stdout; (3) include at least one non-ASCII + one ordering-discriminator row when the projection claims ordering (a ts-ascending fixture cannot distinguish insertion-order from a ts-sort ‚Äî `trg-out199`).
