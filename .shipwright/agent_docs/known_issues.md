## Awaiting-launch state — expected latency band (iterate v0.8.2 AC-5)

When a slash-command launch (`/shipwright-iterate <prompt>` etc.) is dispatched
into the embedded terminal, the task transitions:

```
draft → awaiting_external_start  (CTA clicked, command pasted into pty)
awaiting_external_start → active (server-side `findByUuid` resolves the JSONL)
```

The transition is bounded by two latencies that compound:

1. **Claude's first JSONL write — 5–15 s.** Claude starts its ingestion + schema
   bootstrap before flushing the first event line. This is internal to Claude
   Code and not configurable from webui.
2. **Server poll cadence — 2–5 s.** The transcript-poll endpoint is sequential
   1 s polling on the client, with retry budget on the server. A ~4 s round
   trip is normal under load.

**Total expected wall-clock: up to ~20 s. Anything ≤ 30 s is expected
behavior** and should NOT be treated as a regression.

**Pure Claude (`new-plain`) is excluded** — JSONL-on-first-message is correct
behavior; the task stays `awaiting_external_start` until the user types
something inside Claude.

**If the field reports > 30 s** the most likely cause is an encoded-cwd
mismatch: pty was started under one cwd but `Set-Location`'d before the
launch command pastes, while `task.cwd` still records the original. To
diagnose:

```bash
# server side
SHIPWRIGHT_DEBUG_AWAITING_LAUNCH=1 npm run dev
# Then trigger a launch and look for `[awaiting-launch] miss …` lines.
# Compare the "encodedCwds=…" list against the encoded cwd you expected
# for `task.cwd`. If your task.cwd appears nowhere in the list, the pty's
# effective cwd diverged.
```

## Image-paste path-of-record (iterate v0.8.3 AC-3)

Image pastes split across two distinct paths depending on **where** the
clipboard event is consumed:

1. **Bare shell prompt (PowerShell / cmd / bash, no Claude TUI active):**
   WebUI's DOM `paste` handler in `EmbeddedTerminal.tsx` intercepts the
   event, uploads the blob to `/api/terminal/:taskId/paste-image`, and
   the server persists under
   `<task.cwd>/.shipwright-webui/pastes/img-<ts>-<rand>.png`. The pty
   then receives the shell-quoted absolute path. This is the
   WebUI-controlled path.

2. **Inside Claude Code's TUI session:** Claude Code's CLI handles the
   clipboard itself — bypasses the DOM ClipboardEvent pipeline because
   xterm.js + Claude's TUI use the async `navigator.clipboard.read()` API
   directly. Claude writes the image to its own cache at
   `~/.claude/image-cache/<sessionId>/<n>.png` and emits the path into
   the prompt buffer. WebUI **does not** observe, persist, or sweep
   this cache — it lives entirely under Claude Code's control.

The split is intentional: **WebUI does not attempt to migrate Claude's
cache** to `<task.cwd>/.shipwright-webui/pastes/` because (a) we do not
control Claude Code CLI's clipboard pipeline; (b) hooking Claude's cache
write would be a fragile reverse-engineering surface; (c) Anthropic may
make the cache root configurable in a future Claude Code release —
track upstream as a GitHub issue rather than patching around it locally.

The privacy disclosure footer in `TaskDetailPage.tsx`
(`PrivacyDisclosureFooter`) surfaces both paths to the user so the
divergence is documented in-product.

## Open compliance finding — C1 (design manifest)

`C1 (Spec FR → plan/design coverage)` is **knowingly left failing**, not
silenced. It is a *design-phase* verifier that demands
`.shipwright/designs/design-manifest.md`, the artifact `/shipwright-design`
produces. This project never ran that phase — it was **adopted**, and
`run_config.completed_steps` is `[project, plan, build, test]`.

It passed for months as *"no planning FRs — coverage trivially satisfied"*,
but on a wrong basis: the FR table existed the whole time and the older spec
parser simply did not recognise the adopted table shape. Running today's
parser against the historical specs shows 32 / 32 / 42 FRs on the very dates
the audit reported "no planning FRs". A parser upgrade flipped the check to
HIGH-fail with **no change to this project**.

- **Producer fix** filed upstream: monorepo triage `trg-d26da6f4` — the
  verifier needs a "no design phase ran" skip, mirroring the `scope=library`

## Open upstream gap — skipped tests have nowhere to go

`record_event.py` builds a `work_completed` `tests` block from
`--tests-passed / --tests-total / --tests-new / --tests-modified / --e2e-run`.
There is **no `--tests-skipped`**, and nothing under `shared/scripts` or the
compliance plugin reads `tests.skipped` yet.

Consequence, and the trade-off this project accepted in
`iterate-2026-07-21-compliance-audit-reconcile`: a host-gated skipped test used
to be folded into the collected total, which made `passed < total` on a green
run and caused the audit's D4 to report FRs as landing in a failing build. That
is now corrected to executed counts — but `test_evidence.py` rendered the old
shortfall honestly as `PASS (N skipped)`, so with `passed === total` that
disclosure disappears and the skip count lands in a field no consumer reads.
The information is preserved in the event log; it is simply not rendered yet.

- Filed upstream: monorepo triage `trg-c40210ee` — asks for a `--tests-skipped`
  flag, consumers that render it, and a D4 that keys on failures rather than on
  the `passed`/`total` gap. Same outbox caveat as above.
- Related sharp edge reported in the same item: `apply_amendments` merges
  **shallowly**, so an `event_amended` carrying `fields.tests` replaces the whole
  block and silently drops siblings such as `e2e_run` (which `test_evidence.py`
  uses to classify a work event's layer). This reconciliation hit that defect
  and repaired it; see the guard test's header.
