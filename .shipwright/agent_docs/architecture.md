## Data Flow

Shipwright Command Center is a two-half local stack: a Hono backend on Node 20+ at `:3847` and a Vite-served React 19 frontend at `:5173`, communicating over a JSON HTTP API under `/api`. The Vite dev server proxies `/api` to the Hono port, so both halves run independently and refuse to silently half-start (Vite via `strictPort`, Hono via a bind-error handler).

The load-bearing architectural choice (ADR-034) is the external-launch model: the WebUI spawns no Claude process. Instead, on task creation the server pre-binds a `crypto.randomUUID()` session-id, and the launcher (`core/launcher.ts`) emits a copy-command for the user's shell (PowerShell, cmd.exe, or POSIX) carrying `--session-id <uuid> --name <title>`. The user pastes that command in their own terminal (or VS Code). The Claude CLI then writes its JSONL transcript at `~/.claude/projects/<encoded-cwd>/<uuid>.jsonl`, which the WebUI observes via `core/session-watcher.ts`. Discovery is filename-first (`<uuid>.jsonl`); first-line sessionId is a sanity check.

The transcript endpoint is stateless: clients pass `?fromByte=<offset>&expectFingerprint=<fp>` so multi-tab support works by construction. UTF-8-safe byte slicing cuts on `\n` boundaries only; EBUSY/EPERM/EACCES/ENOENT are retried up to 6 attempts with exponential backoff. Client polling at 1 s replaces SSE вҖ” no chokidar, no heartbeat. The Inbox derives pending tool_use blocks from the same JSONL via `core/inbox-derive.ts` with a `(sessionUuid, mtimeMs, dismissedKey, contentLength)` cache key. **Iterate-2026-05-15 (inbox-awaiting-user):** `inbox-derive.ts` additionally runs `detectAwaitingUserQuestion()` вҖ” a content-only walk that flags a session whose latest turn is a turn-ended assistant message asking the user a plain-text question (text ends with `?` or holds an enumerated option list, no `tool_use` block). `deriveSessionInbox()` unifies both paths and applies the precedence rule in one place (a pending tool_use wins). The `/api/external/inbox` response items gain a `kind: "ask_tool" | "text_question"` discriminant; `text_question` rows auto-clear on the next user turn (no dismiss, no persisted state) вҖ” the mtime cache key is unchanged because detection is pure over already-parsed events. **Iterate-2026-05-18 (inbox-terminal-prompts):** a third detection source is the live `@xterm/headless` mirror вҖ” the `/api/external/inbox` route calls `PtyManager.peekTerminalText` + `core/terminal-prompt-detect.ts extractTerminalPrompt` as a post-pass on every poll (outside the JSONL mtime cache, since live screen state changes without a JSONL mtime change) to surface a waiting `AskUserQuestion` picker, which never reaches the JSONL (Claude Code journals a tool turn only after it returns). The response gains `kind: "terminal_prompt"` (with `promptText`); precedence per task is `ask_tool > terminal_prompt > text_question`. This is the inbox's first read of non-JSONL state вҖ” best-effort, live-mirror-only (external-terminal launches have no mirror and are not covered). Inbox cards also pass `{ focusTerminal: true }` nav-state so TaskDetail lands with the embedded terminal focused.

State lives in three files under `~/.shipwright-webui/`: `projects.json` (registry), `sdk-sessions.json` (task store, schemaVersion v1/v2/v3 with write-on-touch migration per ADR-038/044), and `settings.json`. All three are guarded by `proper-lockfile`; ELOCKED surfaces as HTTP 409. Stack profiles resolve in three steps (`SHIPWRIGHT_PROFILES_DIR` вҶ’ `SHIPWRIGHT_MONOREPO_PATH/shared/profiles` вҶ’ bundled `server/profiles/`).

Project configuration flows in via `<project>/.webui/actions.json` (overlaying the bundled `default-actions.json`) and read-only consumption of `<project>/shipwright_run_config.json` (per `core/run-config-reader.ts`). The WebUI never writes into `~/.claude/projects/` or `shipwright_run_config.json`; the only WebUI write surfaces are `~/.shipwright-webui/*.json`, the `.webui/actions.json` stub creation, the `.webui/actions.json` upload + reset endpoints (FR-01.27), `<project.path>/.shipwright-webui/<slug>.code-workspace` (POST `/api/projects` only вҖ” idempotent VS Code workspace bootstrap, ADR-059), `<task.cwd>/.shipwright-webui/pastes/img-<unix-ms>-<8 hex>.<ext>` (image-paste from the embedded terminal; magic-byte mime sniff, 8 MiB cap, deterministic keep-last-N retention вҖ” ADR-067, path migrated in iterate v0.8.2 AC-6 from the legacy `<task.cwd>/.claude-pastes/` layout) plus an idempotent append of `.shipwright-webui/` to `<task.cwd>/.gitignore` when the user accepts the suggestion toast (a pre-existing legacy `.claude-pastes/` line counts as already-covering, so projects that gitignored the old path don't get a duplicate entry), and вҖ” added by ADR-068-A1 (iterate-2026-05-04) вҖ” `~/.shipwright-webui/terminal-scrollback/<taskId>.log` (rotated to `<taskId>.log.1` at `SHIPWRIGHT_TERMINAL_SCROLLBACK_MAX_BYTES`, default 1 MiB). The scrollback dir is created with mode `0o700` and individual log files with `0o600` (POSIX-enforced; Windows ignores POSIX modes вҖ” privacy disclosure UI surfaces this caveat). UUID-format validation guards every public ScrollbackStore method; realpath-at-op-time on `clear()` and `rotate()` defeats mid-runtime symlink swaps. TTL-based daily sweep (default 1 day) bounded at 100 file groups per pass, oldest-first by mtime, skips active task IDs (sdk-sessions state вҲҲ {active, idle, awaiting_external_start, jsonl_missing} OR live pty entry). All write paths flow through `realPathGuard` to refuse symlink-escape; the upload path additionally rejects oversized payloads via a `Content-Length` pre-check before buffering. The Preview spawn path uses `shell: false` to neutralize command injection. The pty-manager spawn target is whitelisted to shell binaries only (never `claude`) so embedded-terminal hosting stays Plan-D''-conform.

ADR-067 introduces the first WebSocket surface in the codebase: `GET /api/terminal/:taskId/ws` (via `@hono/node-ws`) is the authoritative ensure-or-create entrypoint for the per-task pty owned by `core/terminal/pty-manager.ts`. Writer ownership is bound to the live WS connection identity (cleared on close/error so a new attach can become writer); a 30-min idle ceiling forces pty kill regardless of connection state; outbound is drop-while-saturated when `WebSocket.bufferedAmount` exceeds the configured cap. `POST /api/terminal/:taskId/spawn` is retained as an idempotent prewarm. The Origin gate matches the existing CORS loopback posture вҖ” adding any remote-access mode would require additional auth (explicit non-goal of this iterate).

ADR-068-A1 (iterate-2026-05-04) introduced WS replay-on-attach + per-task disk-scrollback persistence; the protocol then evolved through ADR-088 (server-side `@xterm/headless` mirror behind feature flag), ADR-089 (single `replay_snapshot` WS envelope + flag flip), ADR-087 (Iterate C, 2026-05-12) which retired the legacy chunked-replay path entirely, and finally ADR-092 (Iterate E, 2026-05-12) which closed ADR-091's live-pty re-attach regression. **Today's replay flow:** on every `onOpen`, the server pauses the pty (`pty.pause()`), resolves the replay snapshot via the new `resolveReplaySnapshot(taskId)` helper вҖ” **live mirror wins over disk**: it first calls `ptyManager.serializeMirrorIfLive(taskId)` (returns an in-memory `SnapshotRecord` from the live `@xterm/headless` mirror); if that returns null (no live mirror вҖ” typical for done/exited tasks or post-server-restart), it falls back to `snapshotStore.read(taskId)` (produced by the cleanup-time `finalizeMirrorSnapshot` or the new detach-time `flushMirrorSnapshot`). It then emits a SINGLE `replay_snapshot` envelope `{data, cols, rows, terminalVersion}` вҖ” flushes any buffered live output and resumes the pty. The client writes the snapshot ONCE via `term.reset()` + `term.write(data)` + `term.scrollToBottom()`. **Detach-time persistence (ADR-092):** when the WS detach handler calls `ptyManager.detachAndCount(taskId, conn)` and the post-detach attach count is 0, `flushMirrorSnapshot(taskId)` fires fire-and-forget вҖ” writes the live mirror to disk WITHOUT disposing it (the pty stays alive). This provides server-restart resilience: even if the Hono process restarts between user's navigate-away and navigate-back, the disk fallback path serves the snapshot captured at the last detach. When no snapshot exists (missing file, version-mismatch, or `@xterm/headless` import failure caught by `headless-probe.ts` at boot) the client receives NO replay history вҖ” a deliberate trade-off (blank terminal with live shell). The legacy chunked envelopes `replay_start` / `replay_chunk` / `replay_separator` / `replay_end` are RETIRED; stale-server frames are silently dropped client-side for mid-deploy skew. **Iterate v0.8.2 (AC-7/8/9) extension still active**: when `task.state вҲҲ {done, launch_failed}` the server skips `ptyManager.spawn()` and `attach()` entirely, sends `{type:"ready", role:"reader", replayOnly:true, scrollbackBytes, retentionDays, scrollbackDir}`, replays the snapshot (if any) via the SAME `replay_snapshot` envelope, and closes the WS cleanly вҖ” the client renders a "Session ended" banner. The `ready` envelope still carries `replayOnly`, `scrollbackBytes`, `retentionDays`, `scrollbackDir`; the `scrollback-meta` follow-up envelope still fires post-`bytes()` resolution (the disk-scrollback writer remains alive for `bytes()` accounting + the `clear-scrollback` route, even though its content has no replay consumer). All append-side writes use `fs.appendFileSync` (synchronous, immediately visible). Rotation goes through a 3-state machine (NORMAL вҶ’ ROTATING вҶ’ ROTATION_FLUSH вҶ’ NORMAL) serialized via `p-queue`. ADR-087 also added a one-shot boot wipe of `<scrollbackDir>/*.log*` gated by `<scrollbackDir>/.iterate-c-wiped.marker` (runs once per post-Iterate-C-deploy server lifetime; snapshot files preserved). NEW endpoint `POST /api/terminal/:taskId/clear-scrollback` (loud throws on failure вҶ’ 4xx/5xx) handles destructive history cleanup independently of `POST /close` (now kill-pty-only вҖ” Decision #18). DELETE `/api/external/tasks/:id` cascade-clears BOTH scrollback files AND the cell-state snapshot file (ADR-087 MEDIUM-B1 privacy boundary). A client-side `LaunchCoordinatorContext` (scoped to TaskDetailPage) replaces the old `webui:launch-copied` window-event flow: clicking the Launch CTA dispatches a monotonic `launchToken` into the context, EmbeddedTerminal consumes the token via prompt-readiness handshake (`dataSeenInitially === true && Date.now() - lastPtyDataAt >= 250ms`, hard cap 3s), then sends `commands[shellKind] + "\r"` over the WS вҖ” ZERO clipboard interaction. Cancel paths: reader-role tab, page-unmount, 30s timeout.

Client-side, the virtualized BubbleTranscript persists per-row measurement sizes to `localStorage["webui.virtualizerCache.<sessionUuid>"]` (schemaVersion 1, JSON, capped at 1000 entries) so subsequent visits to a long session avoid the slow-scroll-up estimate-vs-measure cascade (ADR-066). Writes fire on `pagehide`, every 5 s, and React unmount; the cache is pruned to currently-active event keys before write and rehydrated on mount via TanStack Virtual's `initialMeasurementsCache`. On cold cache, a one-frame warmup raises `overscan` so all visible rows mount + measure before the user can scroll. This is browser-local state вҖ” no server involvement.

ADR-101 (iterate-2026-05-14, FR-01.30) adds a Triage Tab + Promote bridge. The webui becomes a read consumer + status-flip producer for `<project>/.shipwright/triage.jsonl` (one JSONL file per registered project). Five new endpoints under `/api/triage/*`: `GET /api/triage/counts` (aggregated `{counts: Record<projectId, number>, total}`); `GET /api/triage/:projectId` (resolved view via TS port `core/triage-store.ts readAllItems` вҖ” drift-protected against `shared/scripts/triage.py read_all_items` via the `server/src/test/fixtures/triage-resolved.json` parity test); `POST /api/triage/:projectId/{promote,dismiss,snooze}`. The TS port mirrors the Python last-status-wins-by-file-order resolution + corrupt-line tolerance. A 5 s mtime-keyed cache fronts the read path; status-write endpoints invalidate the cache for that project. **New write surface:** webui appends `{event:"status",...}` lines to `<project>/.shipwright/triage.jsonl` (lock-order convention: triage.jsonl FIRST, then sdk-sessions.json вҖ” smaller blast radius). Lock primitive is `proper-lockfile` (directory-based); cross-process composition with Python's `_FileLock` (msvcrt/fcntl byte-locks) is documented as a known limitation in ADR-101 вҖ” collision risk is bounded by `triage_promote.py` being a manual-only operator action, and append-mode small writes are line-atomic at OS level. **Promote is a cross-store transaction**: `findByPromotedFromTriageId(triageId)` lookup is the idempotent-retry gate (called twice under the sdk-sessions lock for defense in depth вҖ” closed by an `Promise.all` concurrency test). Status-flip failures after task creation return 207 Partial with the new `taskId` so the client can retry; retry hits the back-ref lookup and reuses the existing task вҖ” no orphan ExternalTasks. Dismiss + Snooze include an orphan-promote guard (409 `promote_in_progress` when a back-ref task exists) so half-promoted items can't be silently dismissed. Sidebar gets a `Triage (N)` badge (orange, distinct from Inbox red) polling `/api/triage/counts` every 30 s with exponential backoff on 5xx. All triage fields render as plain text (no MarkdownText) вҖ” XSS-safety mirror of MasterTaskCard's domain-chip pattern from ADR-100. Per-project failure isolation in counts uses `Promise.allSettled` so a single bad project doesn't fail the badge for all others. New owned files: `server/src/{routes/triage.ts,core/triage-{store,paths,write}.ts,types/triage.ts}`, `client/src/{pages/TriagePage.tsx,components/triage/*,hooks/useTriage.ts,lib/triageApi.ts,components/sidebar/TriageBadge.tsx}`. The new `findByPromotedFromTriageId` finder + `create({...promotedFromTriageId})` argument extend `core/sdk-sessions-store.ts` (additive; ADR-100's `promotedFromTriageId` field declaration is now populated).
