### AC-1 — Scrollback ANSI cursor-code filter (v2 after external review)

**Root cause.** `pty-manager.ts:269–278` calls `scrollbackStore.append(taskId, Buffer.from(data, "utf8"))` UNCONDITIONALLY on every `pty.onData` chunk. Raw bytes include cursor-control sequences (`\e[H` cursor-home, `\e[K` clear-line, `\e[?25l/h` cursor visibility, `\e[<n>;<m>H` absolute positioning, `\e[?47/1049h/l` alt-screen) AND repaint-oriented C0 controls (`\b` backspace, bare `\r` without `\n`). When persisted to disk and replayed via the chunked-envelope flow into a fresh xterm, those re-execute as cursor jumps that overwrite the playback — destroying the visible conversation history.

**Fix.** Introduce a sanitizer at the persistence boundary. The LIVE WS broadcast remains unfiltered (those bytes still drive the live xterm correctly). The DISK PERSIST path runs through the sanitizer. **Read path also runs the sanitizer (per OpenAI #3)** so v0.8.0 scrollback files on disk re-attach cleanly without a backfill migration.

**Where.** Inside `ScrollbackStore` — single bottleneck, keeps `pty-manager` dumb, makes the contract explicit. Sanitizer is a pure function `sanitizeForScrollback(input: Buffer | string, carry: SanitizerCarry): { out: string; carry: SanitizerCarry }` plus `SanitizerCarry` state held per-task in `PerTaskState`.

**Persisted format (after external-review tightening — OpenAI #1).** Define explicitly:
- **Preserve** — printable UTF-8 codepoints, `\n` (LF), `\r\n` (CRLF — preserve the pair as-is), `\t` (HT), all SGR sequences `\e[<params>m` (colors / bold / italic / underline / dim / reverse-video / blink, including `\e[38;5;…m` 256-color and `\e[38;2;r;g;bm` truecolor), OSC sequences `\e]…\x07` or `\e]…\e\\` (window-title etc., rare).
- **Strip** — bare `\r` (CR not followed by LF — repaint trigger), `\b` (backspace — repaint trigger), all CSI sequences with non-`m` final bytes (cursor movement, erase, scroll, alt-screen, save/restore, private `\e[?…h/l`).
- **Drop on parse failure** — incomplete escape sequences over the carry-buffer cap or unrecognized CSI structure: drop the sequence, keep going. Resync on the next safely-parsed byte.

**Implementation strategy (UTF-8-safe + protocol-aware, per Gemini #1 + OpenAI #2).**
- Operate on **raw bytes** for ESC scanning — never decode partial UTF-8. The scanner identifies escape sequences purely by ASCII byte values (ESC = `0x1B`, CSI = `[` = `0x5B`, OSC = `]` = `0x5D`, etc.). Multi-byte UTF-8 codepoints are not affected by the scanner because they cannot contain `0x1B`/`[`/`]` as continuation bytes (UTF-8 continuation bytes are always `0x80–0xBF`).
- Use a per-task state machine: `state ∈ {GROUND, SAW_ESC, IN_CSI, IN_OSC, IN_OSC_ESC}`. On chunk boundary, carry only the raw bytes of an in-progress sequence + state.
- **Carry caps** (per Gemini #5): CSI carry max `32` bytes, OSC carry max `4096` bytes. On overflow, drop the in-progress sequence (do NOT emit fragments) and reset to `GROUND`. The next legitimate `\e` resyncs the parser.
- Final UTF-8 decoding on the **emitted** sanitized byte stream uses `StringDecoder` so codepoints split across chunks round-trip correctly.

**Read path (legacy compat, per OpenAI #3).** `ScrollbackStore.read()` reads the on-disk bytes (which for v0.8.0 files contain raw cursor codes) and runs them through the same sanitizer with a fresh carry. v0.8.1+ files are pre-sanitized, so the second pass is a no-op. CHANGELOG note: legacy scrollback re-attaches cleanly; no migration script needed.

### AC-2 — Black-on-black input rendering (v2 after external review)

**Root cause (hypothesis, diagnosis-first).** Claude Code TUI uses ink + chalk. With our `TERM=dumb FORCE_COLOR=1` (set in `routes.ts:626–632` to neutralize chalk's truecolor branch on Windows), chalk emits 16-color ANSI escapes. The input box likely renders with `\e[40m` background (ANSI black → our theme's `#1a1a1a`) AND `\e[30m` foreground (also `#1a1a1a`) → black-on-black. OR `\e[7m` reverse-video on a dark default produces a similar collision.

**Diagnosis step (test-only, no app-code debug pathway — per OpenAI #14).**
- Run a Playwright test that opens TaskDetail, attaches the WS-frame-capture used by Spec 76, drives a Claude Code launch + a typed-input attempt, and dumps the captured `data` frames to `test-output/ac2-frames.log` for inspection.
- The diagnosis test does NOT ship; it lives under `client/e2e/diagnostic/` (gitignored or `*.diag.ts`-suffixed) and is removed before merge. **No `?debug-bytes=1` query param** or client-side console-dump pathway ships in app code (terminal output may contain secrets / env vars / tokens — diagnostic capture stays test-local).
- Diagnosis output (the offending escape sequence) is recorded verbatim in the iterate ADR body so future regressions have a fixture.

**Option A (theme-extension — preferred per Gemini #4).** If a single ANSI slot collision is identified, remap that slot in the theme block (lines 285–319). Likely candidates:
- Remap ANSI `black` slot from `#1a1a1a` to a slightly-not-black value (e.g. `#2a2a2a`) so even when fg=bg they differ visually. Trade-off: changes the look of any other Claude/PowerShell output that prints black-on-default.
- Or: define a CSS variable `--term-input-bg` that the theme reads, set it to a near-black-but-distinct value.

**Option B (dark-theme switch for Claude Code TUI — fallback only).** Apply once at terminal session start (NOT mid-stream — per Gemini #4) based on a deterministic server-provided session flag (e.g. `task.kind === "new-plain"` already implies Claude Code). Theme switch happens in the `useEffect([])` mount block before xterm renders. Larger change; only used if option A can't address the collision. Documented in ADR.

**Tests (RED first).**
1. `EmbeddedTerminal.test.tsx` — new describe block "Theme contrast":
   - For each (fg-slot, bg-slot) pair where the live theme could render text, assert WCAG AA contrast ratio ≥ 4.5:1
   - Specifically: theme.foreground vs theme.background — must pass
   - Specifically: theme.black vs theme.background — must pass
   - Specifically: theme.foreground vs theme.black — must pass
2. **Fixture-driven assertion (per OpenAI #6) — primary regression guard, not supplemental.** New unit test that takes a recorded byte fixture (the actual escape sequence diagnosed in step 1) and asserts that, when fed into a headless `xterm` instance using the project's theme, the rendered fg/bg pair at the input-box cell location differs by ≥ 4.5:1 contrast.
3. (Optional, deferred if time-tight) Live visual e2e — render the diagnosed fixture in the real xterm container, screenshot the input region, assert non-uniform pixel hue.

**Files.** `client/src/components/terminal/EmbeddedTerminal.tsx` (theme block), `client/src/components/terminal/EmbeddedTerminal.test.tsx`. **Diagnostic Playwright spec** under `client/e2e/diagnostic/ac2-claude-tui-bytes.diag.ts` — local-only, removed pre-merge.

#### 3a — pause/resume refcount with per-conn ownership (`pty-manager.ts`)

**Root cause.** `pty-manager.ts:349–374` exposes `pause(taskId)` / `resume(taskId)` that fire `pty.pause()` / `pty.resume()` unconditionally. With multiple WS connections each running replay-on-attach (multi-tab), conn-B's `pause()` is a no-op (already paused) but its `resume()` un-pauses for conn-A too — conn-A's replay is interrupted by live data.

**Fix (per Gemini #3 + OpenAI #12 — keep per-conn ownership).**

Replace the token-less API with **per-conn pause ownership**:

```ts
// New API:
pauseForConn(taskId, connToken): increments per-task refcount AND records the conn
                                  in a per-task pausedConns: Set<connToken>;
                                  fires pty.pause() only on 0→1 transition.
resumeForConn(taskId, connToken): if connToken ∈ pausedConns, decrement +
                                   delete from set; fires pty.resume() only on 1→0.
                                   Idempotent — no-op if conn isn't pausing.
```

Plus: `detach(taskId, connToken)` calls `resumeForConn(taskId, connToken)` BEFORE freeing the connSubs entry. This guarantees that ANY exit path (graceful close, watchdog forced eviction, server shutdown) cleans up that conn's pause stake.

`routes.ts` already calls `ptyManager.pause(taskId)` / `ptyManager.resume(taskId)` at lines 477 + 554. Migrate these call sites to `pauseForConn(taskId, connToken)` / `resumeForConn(taskId, connToken)` — `connToken` is already in scope. The old token-less methods stay (deprecated, internal) for backwards-compat in tests.

**Tests.**
- Refcount semantics: 2 pauseForConn() with different tokens → 1 pty.pause(); resumeForConn() for both → 1 pty.resume(); imbalanced (resume without prior pause for that token) is a no-op
- Multi-conn scenario: conn-A pauses, conn-B pauses, conn-A resumes (pty stays paused), conn-B resumes (pty resumes once)
- **Forced-eviction cleanup**: pause for conn-A, then call `detach(taskId, connA)` directly (simulates watchdog) — pty.resume() fires, refcount returns to 0, no leak

#### 3b — Writer-slot release under stuck connection (`pty-manager.ts`)

**Root cause.** `pty-manager.ts:468–496` (detach) only fires on ws.close. Under high pty-output volume, ws.close events get delayed in the Hono/@hono-node-ws scheduling queue (the close envelope waits behind the data envelopes that haven't drained yet). The new tab attaches as `reader` and stays a reader for seconds-to-minutes — the read-only banner appears even though the prior tab is dead/navigated-away.

**Fix v2 (per Gemini #2 + OpenAI #9 — track socket drainage, NOT pty emission).**

The HIGH-severity finding from both reviewers is that the original heuristic (`bufferedAmount > threshold` AND `pty.lastDataAt > 2s`) is INVERTED: a runaway pty keeps `lastDataAt` fresh, so a stuck writer is never evicted under exactly the load conditions that cause the bug.

Replacement heuristic — **track per-conn socket drainage**:

```ts
interface PtyEntry { …;
  // Per-conn tracking. Key is connToken (not entry-level).
  bufferedExceededSince: Map<unknown, number | null>;  // ms timestamp when bufferedAmount first crossed threshold; null = below
}
```

In `deliverWithBackpressure()` (already runs on every outbound chunk):
- After computing `live = (conn as {bufferedAmount?: number}).bufferedAmount`, check capability (see below).
- If `live > STUCK_THRESHOLD_BYTES` and `bufferedExceededSince.get(conn) == null`: record `Date.now()`.
- If `live <= STUCK_THRESHOLD_BYTES` and `bufferedExceededSince.get(conn) != null`: set null (drained — clear the timer).

Watchdog tick (every 2s):
- For each entry with `entry.writer != null`:
- Read `since = entry.bufferedExceededSince.get(entry.writer)`.
- If `since != null && Date.now() - since >= STUCK_DURATION_MS`: this writer's socket has been backed up for >= 2s — evict via `this.detach(taskId, entry.writer)`. The detach path runs the per-conn resume cleanup (per 3a), promotes next reader, fires `onPromoteToWriter`.

**Defaults** (tunable via constructor opts):
- `STUCK_THRESHOLD_BYTES = wsBufferBytes / 2 = 524288` (half the per-conn cap)
- `STUCK_DURATION_MS = 2000`
- `WATCHDOG_INTERVAL_MS = 2000`

**Capability check (per OpenAI #10).** Before relying on `bufferedAmount`, probe at construction-time: spawn a placeholder pty + fake-conn, check `typeof conn.bufferedAmount === "number"`. If undefined, log a one-time warning and disable the watchdog (the fallback is still better than v0.8.0 — the legitimate-close path remains, just no recovery from stuck-close). For the @hono/node-ws WSContext we also fold a defensive `??= 0` so `null` doesn't break math. Manual smoke check during build verifies the live ws.bufferedAmount path emits real numbers.

**Writer epoch token (per OpenAI #11).** Add `entry.writerEpoch: number` (monotonic counter, increments on every writer change including watchdog evictions). When a delayed `ws.close` arrives later for an evicted conn, `detach()` checks `entry.writer === conn` first (already does) — the slot has already moved, so the stale close is a no-op for the writer slot. (Existing logic; just verified covered by tests.)

**Tests.**
- `pty-manager.test.ts` watchdog suite:
  - **Healthy under load**: writer with `bufferedAmount = 0` continuously through the watchdog window — no eviction
  - **Healthy under burst**: writer with `bufferedAmount` briefly > threshold but drains in <2s — no eviction (because `bufferedExceededSince` resets to null)
  - **Stuck**: writer with `bufferedAmount > threshold` and never drains for ≥ 2s — evicted; `detach()` runs; reader promoted; `onPromoteToWriter` fired
  - **Capability missing**: construct PtyManager with FakePty whose conn has no `bufferedAmount` field → watchdog logs warning, never evicts (graceful degradation)
  - **Force-eviction releases pause refcount**: pre-pause via `pauseForConn(taskId, connA)`, then watchdog evicts connA → `pty.resume()` fires (verified by FakePty `__paused === false` post-eviction)
  - **Stale close after eviction**: evict connA via watchdog, then call `detach(taskId, connA)` (simulates the delayed close finally arriving) → no error, writer slot already moved to connB stays
  - **Watchdog timer cleanup on `killAll()`**: no leftover interval handle (tested via `vi.useFakeTimers()` + `expect(vi.getTimerCount()).toBe(0)` post-killAll)
  - **Watchdog timer cleanup on individual `kill(taskId)`**: same as above for a single task
- `client/e2e/flows/78-reattach-under-pty-load.spec.ts` (NEW) — using Spec 76 WS-frame-capture pattern:
  - Launch task, drive high-volume PowerShell payload (`for ($i=1; $i -le 5000; $i++) { "line $i" }`)
  - Close the page mid-flow (Playwright `page.close()`) → open a new page → re-attach to the same taskId
  - Capture WS frames on the new page; assert `ready{role:writer}` arrives within 5s (i.e. the watchdog evicted the stale writer)
  - Assert no `read_only` envelope received after the writer-promoted envelope

**Files.** `server/src/terminal/pty-manager.ts` (refcount + watchdog + capability check + epoch), `server/src/terminal/routes.ts` (migrate `pause`/`resume` call sites to `pauseForConn`/`resumeForConn`), `server/src/terminal/pty-manager.test.ts` (audit existing + add watchdog suite), `client/e2e/flows/78-reattach-under-pty-load.spec.ts` (NEW).

**Risk.** Medium-High. Watchdog timing is delicate. The v2 heuristic (per-conn drainage, not pty emission) closes the inverted-logic bug both reviewers caught. Capability check + writer epoch make the heuristic gracefully-degradable. Initial defaults conservative (2s grace, 512 KiB threshold).

## Alternative considered (rejected)

**Run all 3 ACs as 3 separate iterates within a campaign.** Pro: each AC commits + pushes independently; tighter blast radius per change. Con: same external-review cost × 3, same finalization cost × 3, three PRs to coordinate.
