## Acceptance Criteria

### AC-1 (Stage 0) — Visual padding inside the dark terminal canvas

User-reported: "Padding → nun haben wir Padding, aber die Schrift des Terminals ist direkt am Rand des schwarzen Bereichs." Currently the v0.8.3 outer `p-2` puts 8px between the tab-panel edge and the start of the black canvas. xterm fills the inner area edge-to-edge → text starts at col 0 = right at the dark edge.

**Goal:** Black bg extends to the wrapper edge (no outer ring vs the parent surface), AND text/cursor/output sits ~8px inset from the black edge on all four sides.

**Implementation:** Single-layer wrapper. Drop the v0.8.3 outer `p-2`; replace with `bg-[#1a1a1a] rounded-md p-2` directly on the EmbeddedTerminal outer `<div>`. xterm's `containerRef` mounts inside; FitAddon resizes to the padded inner box; `cols` may shrink by 1-2 characters in the worst case (already accepted in v0.8.3 spec). The black extends edge-to-edge of the wrapper; text starts after the inner 8px ring of bg.

**Conditional banners** (`readOnly`, `replayOnly`, `previewCommand`) currently sit OUTSIDE the dark area as `border-b` rows above the canvas. With the new single-wrapper they end up INSIDE the dark padded box; their light-bg fills `bg-[var(--color-warning-bg,#fff7ed)]` etc. would clash with the surrounding dark padding ring. Tweak: extend the banner classes to span full width by negating the wrapper padding (`-mx-2 -mt-2 mb-2`) so they read as "header strip" inside the dark frame, OR move them above the wrapper. **Decision during build:** prefer `-m` negation (keeps the banners visually attached to the terminal context).

**PrivacyDisclosureFooter** is `absolute bottom-2 left-2 right-2` from v0.8.3 AC-2 — those coordinates already point at the wrapper's content-area edges; no change expected. Verify visually during smoke.

**Files:** `client/src/components/terminal/EmbeddedTerminal.tsx` (wrapper className + banner negative-margin).

**Severity: LOW.** **Estimate: 0.25d.**

### AC-2 (Stage 1) — Remove the v0.8.3 Ctrl+V handler

User-reported: "CTRL V geht immer noch nicht. Wir nehmen das raus. Das ist mir zu viel Theater. ALT V geht super." Empirically the v0.8.3 `attachCustomKeyEventHandler` + `clipboard.read()` flow doesn't work in the user's daily flow. Alt+V (Claude TUI's own clipboard pipeline) is sufficient and lands the image under `~/.claude/image-cache/`.

**Goal:** Remove all v0.8.3-AC-1 code — the Ctrl+V keydown interception. KEEP:
- The DOM `paste` event handler in `EmbeddedTerminal.tsx` (still useful for right-click → Paste menu and programmatic paste flows).
- The shared `uploadPasteBlob` helper (used by the surviving `paste` handler).
- The `/api/terminal/:taskId/paste-image` route + the v0.8.2 latency optimization.

**Remove:**
- `client/src/components/terminal/clipboard-paste.ts` (helper module — no remaining caller).
- `client/src/components/terminal/clipboard-paste.test.ts` (tests for the removed helpers).
- `term.attachCustomKeyEventHandler(...)` registration in EmbeddedTerminal.tsx.
- `ctrlVHandlerRef` + the useEffect that updates it.
- `shouldInterceptCtrlV` / `readClipboardForPaste` imports.
- The `Ctrl+V real-fix wiring` describe block (5 tests) in `EmbeddedTerminal.test.tsx`.
- `client/e2e/flows/80-ctrl-v-real-paste.spec.ts` (Spec 80 e2e).
- The xterm mock's `attachCustomKeyEventHandler` spy + `registeredKeyHandler` capture (no longer needed).

**Files:** `client/src/components/terminal/EmbeddedTerminal.tsx`, `client/src/components/terminal/EmbeddedTerminal.test.tsx`, `client/src/components/terminal/clipboard-paste.ts` (delete), `client/src/components/terminal/clipboard-paste.test.ts` (delete), `client/e2e/flows/80-ctrl-v-real-paste.spec.ts` (delete).

**Severity: MEDIUM** (user-visible expectation cleanup). **Estimate: 0.5d.**

### AC-3 (Stage 2) — Scrollback re-replay no longer duplicates Claude banner on tab revisit

User-reported: "Wenn ich zur Session zurückgehe habe ich gefühlt 100x den 'Claude Start' header im Terminal drin. Beim Launch nur einer." Hypothesis: WS re-attach replays the FULL disk-backed scrollback every time, AND xterm doesn't clear before replay → after N tab-flips the user sees N copies of the historical session.

**Investigation steps (Stage 2 build-phase BEFORE writing code):**
1. Confirm hypothesis by triggering a clean re-attach with `SHIPWRIGHT_DEBUG_REPLAY=1` (or by inspecting `server/src/terminal/routes.ts` WS upgrade handler) — count `replay_chunk` envelopes per attach + verify whether xterm receives `\x1bc` (RIS reset) before each replay.
2. Determine where the dedupe should live: server (only replay on first attach within a session, not on every WS upgrade) OR client (xterm.reset() before consuming replay envelopes) OR the `replay_separator` envelope semantic gets honoured by `useTerminalSocket` to clear the buffer.

**Acceptance:** After tab-flip TaskBoard → Task → TaskBoard → Task (3 round trips), the user sees exactly ONE Claude welcome banner in the terminal. The historical scrollback is preserved (per ADR-068-A1) but rendered exactly once.

**Files (best-guess, may grow during investigation):** `server/src/terminal/routes.ts` (WS upgrade replay path), `client/src/hooks/useTerminalSocket.ts` (replay-envelope handling), `client/src/components/terminal/EmbeddedTerminal.tsx` (xterm reset on socket reconnect, possibly).

**Severity: HIGH** (visual breakage on every revisit; affects UX confidence). **Estimate: 0.5-1d.**

### AC-4 (Stage 3) — `awaiting_external_start` reliably transitions to `active`

User-reported: "Awaiting Launch immer noch da. Da funktioniert was nicht. Geht nicht auf active." The task stays in `awaiting_external_start` forever after Launch, even though Claude is plainly running in the terminal pane.

**Investigation steps (Stage 3 build-phase BEFORE writing code):**
1. Restart the server with `SHIPWRIGHT_DEBUG_AWAITING_LAUNCH=1` per `known_issues.md` instructions.
2. Trigger a fresh Launch and capture the `[awaiting-launch] miss …` log lines.
3. Diagnose:
   - **encoded-cwd mismatch:** task.cwd encodes to `<X>` but Claude wrote JSONL into `<Y>` because pty cwd diverged → fix by re-encoding task.cwd at attach-time, or pinning Claude's cwd more strictly.
   - **filename-first discovery miss:** session-watcher expects `<sessionUuid>.jsonl` but the file is named differently → check session-watcher mod
   - **JSONL never written:** Claude failed to start (still possible despite hook spam being non-blocking) → diagnose Claude-side.
4. After diagnosis, write a TARGETED fix matching the root cause + a regression test.

**Acceptance:** A fresh Launch transitions `awaiting_external_start` → `active` within 30s wall-clock as documented in `known_issues.md` ("Total expected wall-clock: up to ~20s. Anything ≤ 30s is expected behavior").

**Files (best-guess):** `server/src/core/session-watcher.ts` (filename-first discovery + retry budget), `server/src/external/routes.ts` (state transition logic), `server/src/core/sdk-sessions-store.ts` (state-machine).

**Severity: HIGH** (core flow broken). **Estimate: 0.5-1d.**

### AC-6 (Stage 1) — Remove the "Terminal" header CTA

User-reported (post-v0.8.4 UAT): "Terminal Button kann raus. Den brauchen wir nicht 'nur um das tab zu wechseln.'" The v0.8.3 ADR-068-A1 AC-16 / Phase-5-Codex review installed a "Terminal" CTA in the TaskDetailHeader that — for `awaiting_external_start` / `active` tasks — just dispatched `webui:focus-terminal-tab` (pure tab-flip, no auto-execute). The `webui:focus-terminal-tab` event listener inside `TaskDetailPage.tsx` becomes dead code (no dispatcher remains) — remove it too. The `…` overflow menu (`Stop terminal session` / `Clear terminal history`) stays unchanged.

**Files:** `client/src/components/external/TaskDetailHeader.tsx` (CTA matrix), `client/src/pages/TaskDetailPage.tsx` (remove the `webui:focus-terminal-tab` listener), test that exercises the CTA matrix (location TBD during build — likely `TaskDetailHeader.test.tsx` if it exists, otherwise spec it during AC-6 build phase).

AC-4 fixes the underlying transition; if it slips, the gap remains. Track as an open follow-up after AC-4 lands.
- **Alt+V latency (was ~4s in v0.8.3 spec, briefly 15s pre-PATH-fix, now 2s after `claude doctor` / restart).** User confirmed resolved post-restart. Out of scope.
- **`claude /doctor` "31 skill descriptions dropped" warning** (Punkt 4). Claude Code state, not WebUI state. Mention in iterate ADR as a known quirk, but no code change.
- **Refactor of the LaunchCoordinatorContext.** State machine is correct; only the scrollback re-replay bug (AC-3) needs fixing. Don't extend.

## Lessons from v0.8.4 to apply

1. **PATH inheritance from a stale harness blocks every subprocess.** The "uv: command not found" hook spam was 100% PATH; restart of Claude Code (the parent) fixed it without code change. Lesson: when "everything restarted" still shows old behavior, look at child-process env first, not at code.
2. **Synthetic-event tests don't prove keyboard paths.** v0.8.3 Spec 79 AC-3 test passed; real Ctrl+V never worked. Spec 80 (real-browser) was the truth-test that should have led the iterate, not followed it. v0.8.5 AC-2 reverts the v0.8.3 fix because Spec 80 in production never gave the user value — Alt+V was always sufficient.
3. **Disk-backed scrollback re-replay is a foot-gun.** Every WS upgrade triggers replay; the client has no built-in dedupe. Without an explicit reset/markup, repeat-attach amplifies the visual N times. Investigate before fixing — the cure is on either the server (don't replay on re-attach within the same task) or client (xterm.clear before replay) or both.
