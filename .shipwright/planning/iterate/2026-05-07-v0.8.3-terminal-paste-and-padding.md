### Stage 1 — Ctrl+V real fix (the v0.8.2 AC-3 was a false positive)

- [ ] **AC-1 (Ctrl+V triggers our DOM paste handler with image clipboard payload):** v0.8.2 moved the paste listener to `document` capture phase but did NOT solve the root cause: xterm.js handles Ctrl+V internally via `navigator.clipboard.readText()` which **bypasses the DOM ClipboardEvent pipeline entirely**. The synthetic-event Spec 79 AC-2/3 test passed because it dispatched a hand-built `paste` event, never crossing xterm's keyboard handler. Real Ctrl+V never fires our listener.

  Real fix: install `term.attachCustomKeyEventHandler(ev => …)` that returns `false` for `Ctrl+V` events, suppressing xterm's default handling. Then either:

  **(a)** Let the browser's native paste handling fire on the helper textarea — works on Chrome/Edge for text payloads but on Windows the OS clipboard often strips images for textarea-targeted pastes (image-paste is unreliable here).

  **(b)** *RECOMMENDED:* On Ctrl+V keydown, programmatically invoke `await navigator.clipboard.read()`, walk the `ClipboardItem.types`, look for `image/*`, get the blob via `clipboardItem.getType(type)`, then route through the same `/api/terminal/:taskId/paste-image` upload path as the existing `<paste>`-event handler. Falls back to text via `clipboardItem.getType("text/plain")` when no image is present.

  Implementation outline:
  - In `EmbeddedTerminal.tsx` after `term.open(container)`, add `term.attachCustomKeyEventHandler(handleKeyEvent)` where `handleKeyEvent` returns `false` for `(ev.ctrlKey && ev.key.toLowerCase() === "v" && !ev.altKey && !ev.shiftKey)`.
  - Add a separate `keydown` listener on the container (capture phase) that fires the same condition, calls `navigator.clipboard.read()`, walks items, extracts image-or-text, and routes through the existing image-upload helper or `socket.send({type:"data", payload: text})`.
  - Surface clipboard-read errors via `onPasteImageError` (same toast surface as the v0.8.0 path).

  Browser permission: Chrome/Edge require user gesture for `clipboard.read()`. The Ctrl+V keypress itself counts as a gesture. No extra permission prompt expected on Chrome/Edge if the page has been interacted with. Firefox does not support `clipboard.read()` for non-text — fall through to xterm's default (text-only paste).

  New e2e regression: Spec 80 (NEW) — real-browser Ctrl+V test using Playwright's `context.grantPermissions(["clipboard-read", "clipboard-write"])` + a synthetic image written to clipboard via `page.evaluate(() => navigator.clipboard.write([new ClipboardItem({"image/png": blob})]))`. Then `page.keyboard.press("Control+V")` and assert that `/paste-image` was hit + the file landed under `<task.cwd>/.shipwright-webui/pastes/`.

  **Severity: HIGH** — the v0.8.2 fix is a false positive; users still cannot Ctrl+V images at the shell prompt.
  **Estimate: 0.5d.**
  **Files:** `client/src/components/terminal/EmbeddedTerminal.tsx` (key handler + clipboard.read flow), `client/src/components/terminal/EmbeddedTerminal.test.tsx` (new unit test for the key handler decision tree), `client/e2e/flows/80-ctrl-v-real-paste.spec.ts` (NEW — real-browser regression).

## Out of scope

- **Migrating Claude Code's image-cache to `<task.cwd>`.** Same reason — out of WebUI's surface.

## Lessons from v0.8.2 to apply

1. **Synthetic ClipboardEvent dispatch in unit tests does NOT prove the real-browser keyboard path works.** Spec 79's AC-2/3 tests passed but the Ctrl+V root cause was untouched. Lesson encoded: every keyboard-shortcut AC needs a Playwright test that uses `page.keyboard.press()` against a real headed-or-headless browser, not `dispatchEvent`.
2. **xterm.js's clipboard handling uses async Clipboard API, not DOM ClipboardEvent.** This is now in `feedback_…` memory candidate territory — worth saving so future iterates don't repeat the wrong assumption. Save as `feedback_xterm_clipboard_async_api.md` after merge.
3. **Image-paste path-of-record is split** between Claude Code (TUI session) and WebUI (shell prompt). The split is a feature, not a bug — document it openly in the disclosure footer + known_issues.md.
4. **Live UAT is mandatory after AC-3 / paste-related fixes.** Server-side wall-clock e2e cannot detect a client-side keyboard-handling regression.
