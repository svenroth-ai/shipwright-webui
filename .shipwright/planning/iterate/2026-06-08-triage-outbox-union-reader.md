- **run_id:** `iterate-2026-06-08-triage-outbox-union-reader`
- **Intent:** CHANGE (Spec Impact: MODIFY — extends the existing triage read/write contract)
- **Complexity:** medium (locked)
- **Risk flags:** `touches_io_boundary` (`.shipwright/triage.outbox.jsonl` JSONL buffer; JSON parse/serialize producer↔consumer) → Boundary Probe + round-trip test + Confidence Calibration mandatory. - **FR:** FR-01.30 (Triage Tab) MODIFY. Relates to ADR-101 / ADR-106 / ADR-110.

## Think Before Coding (Karpathy)

**Problem.** D1 moved background ("idle-main") triage producers off the git-tracked `.shipwright/triage.jsonl` and onto a per-tree, **gitignored** buffer `.shipwright/triage.outbox.jsonl` (no header — it is a buffer, not a store). The D2 sweep folds that buffer into the next iterate PR branch. The webui server reads **only** the tracked file (`core/triage-store.ts readAllItems`), so the live Inbox now misses every background finding until a sweep+merge round-trips it back into tracked main — a regression (findings can be invisible for hours/days).

**Empirically confirmed live:** the plugin cache `~/.claude/plugins/cache/shipwright/shared/` is synced to D1–D3 (`triage.py` two-pass union reader + `OUTBOX_FILE`; gitignore template + `gitignore_selfheal.py`; `setup_iterate_worktree.py` sweep+self-heal). An empty `.shipwright/triage.outbox.jsonl` already exists on the webui main tree → producers are already routing there.

**Decision.** Mirror the Python contract verbatim on the TS surface, **without changing any public signatures**, so the route layer inherits the behavior with zero churn:
1. `readAllItems(trackedPath)` reads the **union** tracked ∪ outbox and resolves with the **two-pass** algorithm D1 introduced (Pass 1 = all `append` events; Pass 2 = all `status` events ordered by `(ts, file-order)`), matching `shared/scripts/triage.py read_all_items`.
2. `appendStatusEvent({jsonlPath})` becomes **residence-derived** (mirrors `triage.py mark_status`, TRACKED-PREFERRED): write the status to the outbox iff the item's `append` lives **only** in the outbox; otherwise to tracked (bootstrapping the header when the id lives nowhere). Without this, a webui dismiss/promote of a background finding would write an orphan status into tracked main — re-introducing exactly the main-tree drift the whole campaign eliminated.

**Alternatives considered.**
- *Reader-only (no residence writer).* Rejected: functionally surfaces items but every webui mutation of an outbox finding pollutes tracked main with drift — defeats the campaign's invariant. A perfectionist reviewer would flag the half-fix.
- *Relax the route's `existsSync(tracked)` guard + lock target so outbox-only-no-tracked items are mutable.* Deferred (documented limitation). It requires editing `routes/triage.ts`, which is anti-ratchet-baselined at 763 LOC; growing it for an extreme edge (a project whose first-ever triage write was a background producer on idle main, before any tracked write) would itself be a bloat violation needing an exception ADR. Keeping signatures stable means the route is untouched and the common case (tracked exists — incl. this repo and every active repo) is fully fixed.

## Acceptance Criteria

- **AC1 — Union read.** `readAllItems(trackedPath)` returns items present in tracked, in outbox, or both; an outbox-only `append` is visible.
- **AC2 — Two-pass cross-file resolution.** A `status` flip in tracked applies to an `append` in the outbox (and vice-versa); cross-file status ordering is by `(ts, file-order)` (chronologically-later wins; file order is the stable tiebreaker, tracked-before-outbox); a same-id `append` in both files (post-sweep/pre-GC) collapses to one item; a `status` for an id absent from both is skipped.
- **AC3 — Python parity (union).** TS union read equals `triage.py read_all_items` byte-for-byte on a staged tracked+outbox fixture (new `triage-union.outbox.jsonl` + `triage-union-resolved.json`, generated by the extended `regen-triage-fixtures.py`). The existing single-file parity gate still passes.
- **AC4 — Residence-derived write.** `appendStatusEvent` writes the status line to the **outbox** when the id's `append` is outbox-only; to **tracked** when it is in tracked (or both); to **tracked** (with header bootstrap) when the id is in neither. The non-target file is left untouched.
- **AC5 — Regression fence.** Existing single-file behavior is unchanged: corrupt-line tolerance, header-only → `[]`, missing → `[]`, status overlay fields, `promotedTaskId` override, unknown-id skip, and the mtime cache returning the same array reference within TTL (now keyed on both tracked+outbox mtimes; an outbox change invalidates).
- **AC6 — Propagation (Codex Q6).** The webui `.gitignore` carries the canonical `/.shipwright/triage.outbox.jsonl` ignore line (landed + committed via `setup_iterate_worktree` self-heal); plugin-cache shared scripts verified synced to D1–D3.

## Known limitation (documented, by design)

Promote/dismiss/snooze of an item that lives **only** in the outbox with **no** tracked `triage.jsonl` file returns `404` (the item is still visible in the Inbox). The route's existence-guard + `proper-lockfile` target are the tracked path; relaxing them would ratchet `routes/triage.ts` past its anti-ratchet ceiling for an extreme edge. Every active repo has a tracked `triage.jsonl`, so outbox-item mutation works end-to-end there via residence-derivation.

## Affected Boundaries

- `.shipwright/triage.outbox.jsonl` — NEW read source (always) + conditional write target (residence-derived). Headerless JSONL buffer.
- `.shipwright/triage.jsonl` — existing tracked store (unchanged format).
- Cross-process wire-format parity with `shared/scripts/triage.py` (producers write the outbox; webui reads it and may write status flips to it).
