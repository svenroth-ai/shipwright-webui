Each AC has an authored AND run Playwright regression case in Spec 83. Ships as **v0.8.7**.

## Acceptance Criteria

### AC-1 (Stage 0) — `new-plain` `active → idle` when pty is gone

User-reported (PUNKT 3 in handover): Übernacht-revisit of a `new-plain` task shows badge stuck on "In progress" / `active`, no Resume CTA in header (header CTA matrix per v0.8.5 AC-6: `idle → Resume`, anything else → no CTA).

**Root cause** (verified via code-read of [server/src/external/routes.ts:843-848](../../../server/src/external/routes.ts#L843-L848)):
The `result.status === "missing"` branch returns early. For `new-plain`, JSONL is never written (per `known_issues.md` "Awaiting-launch state — expected latency band"), so `firstJsonlObservedAt` stays `null`, so the existing `active → idle` transition at L877 is unreachable.

**Goal:** When the transcript-poll endpoint runs for a `new-plain` task in `state=active` AND `ptyManager.get(taskId) === undefined` (pty was killed by idle-ceiling, server-restart, /close, or DELETE cascade), patch state to `idle`. Self-healing: AC-4 from v0.8.5 will re-flip to `active` on next WS attach.

**Per external plan review (2026-05-08) — wiring + idempotency:**
- `ptyManager` is a **REQUIRED** parameter to `createExternalRoutes()` (NOT optional). Production call site in `server/src/index.ts` passes the singleton; existing tests get a stub with `get: () => undefined`.
- Patch is idempotent by construction: only fires when `task.state === "active"`. Multiple polls after the first patch see `state === "idle"` and the conditional short-circuits — no flapping.
- Auth posture: WebUI is loopback-only by default; `HONO_HOST=true` is gated by `WEBUI_TRUSTED_ORIGINS`. The `transcript` endpoint is becoming side-effecting (`store.patch + store.persist`), but the existing access-gate is sufficient (no new exposure surface).
- No grace-window guard added — `ptyManager.entries.get()` is a synchronous Map lookup; spawn is synchronous from WS-upgrade onOpen, so there is no in-progress-spawn race. The existing AC-4 contract (`onOpen → state=active`) IS the guard against premature idle.

**Acceptance:**
- (E) Given a `new-plain` task in `state=active` with no live pty, when the transcript-poll endpoint is queried, then state patches to `idle` within one poll cycle.
- (E) Given the same task, when the user re-attaches via WS upgrade, then v0.8.5 AC-4 flips state back to `active` (verified by integration with existing AC-4 contract).
- (E) Given a `slash-command-launch` task in same state shape (no pty), the same code path does NOT flip its state (only `actionId === "new-plain"` gate fires).
- (U) Given the patch fired and state is now `idle`, when transcript-poll is called again for the same task, then NO additional patch is emitted (idempotency: the `task.state === "active"` guard short-circuits).
- (U) Given `createExternalRoutes` is called WITHOUT `ptyManager`, then construction throws (compile-time TS error AND runtime guard).

### AC-2 (Stage 1) — Shell-stopped visual marker on every pty-kill

User-reported (PUNKT 2): scrollback contains many PowerShell-startup banners interspersed with empty CRLF blocks; on overnight-revisit the boundary between historical "shell lifetimes" is opaque.

**Goal:** Every pty kill (idle-ceiling, /close, DELETE cascade, killAll on SIGTERM) writes exactly ONE dim-grey ANSI marker frame to the disk-scrollback. Marker is appended via `scrollbackStore.append()` (same path as live pty.onData), so it's persisted, survives WS reconnect, and replays correctly to xterm. Marker is best-effort — failures are logged but never throw.

```
\r\n\x1b[2m──── shell stopped at HH:MM:SS ────\x1b[m\r\n
```

**Per external plan review (2026-05-08):**
- **Timing:** marker is appended INSIDE the existing `pty.onExit` handler at [pty-manager.ts:377](../../../server/src/terminal/pty-manager.ts#L377) — fires AFTER all dying-process flush bytes have arrived. Avoids the gemini-flagged race where `pty.kill()` precedes terminal-state-flush bytes that would render below our marker.
- **Idempotency:** new field `entry.closing = false`. The `kill(taskId)` and idle-ceiling paths set `closing = true` before invoking `entry.pty.kill()`. The `onExit` handler appends the marker only if `entry.closing === true` (avoids 2 markers if onExit fires twice; avoids 1 marker for a shell that crashed naturally — that case is out-of-scope per spec).

**Acceptance:**
- (E) Given a live pty for a task, when `ptyManager.kill(taskId)` is invoked, then the disk-scrollback file gains exactly one marker line containing the literal `──── shell stopped at` string after `pty.onExit` fires.
- (E) Given the idle-ceiling timeout fires, then the same marker is written exactly once.
- (E) Given `kill(taskId)` is called twice rapidly for the same task, then exactly one marker is written (closing-flag dedupe).
- (E) Given the shell exits naturally (`pty.onExit` fires WITHOUT a `closing` flag set), then NO marker is written.
- (E) Given a kill→respawn cycle within one WS attach session, then the next replay-on-attach contains the marker between the historical and the new shell-startup output.
- (E) The marker is rendered by xterm with dim-grey styling (visible via `term.buffer.active` inspection).
- (E) `scrollbackBytes` reported on `scrollback-meta` envelope reflects the marker bytes (multibyte UTF-8 box-drawing chars accounted correctly).

### AC-3 (Stage 2) — Replay-time collapse of PowerShell-startup boilerplate

**Goal:** When the scrollback file is read **for replay**, post-process the read buffer to detect anchored repeats of the literal "PowerShell N.N.N\r\nPS …>" boilerplate AND surrounding 40+-line CRLF runs. Collapse multi-banner-bursts (≥2 banner+CRLF-block sequences within a single shell-lifetime span) to ONE banner + ONE marker line. Disk file is unmodified — only the bytes emitted on replay change.

**Per external plan review (2026-05-08) — architectural separation:**
- **NEW method `bytesForReplay(taskId): Buffer`** in `scrollback-store.ts`. The existing `bytes(taskId)` stays RAW so `scrollback-meta` envelope size accounting + privacy-disclosure copy stay correct.
- WS replay handler at `terminal/routes.ts` switches to `bytesForReplay()` for the `replay_chunk` payload only. The `scrollbackBytes` reported in `ready` envelope keeps reading `bytes()` length.
- Collapse never crosses an AC-2 `──── shell stopped at ────` marker — the regex anchors INSIDE a single span (split by markers, collapse per-span, rejoin).

**Bounded-regex design (no ReDoS):**
- OSC title-set sub-pattern: `(?:\x1b\]0;[^\a]{0,256}\a)?` — bounded `{0,256}` instead of `[^]*?` lazy. If a malformed OSC has >256 bytes without BEL, the pattern fails to match (graceful — no collapse).
- Banner anchor: `(?:\x1b\]0;[^\a]{0,256}\a)?PowerShell \d+\.\d+\.\d+\r\n(?:PS [^\r\n>]{0,512}>\s*)?` — bounded prompt path.
- Trigger gate: requires ≥10 consecutive `\r\n` immediately preceding the banner anchor — the post-resize CRLF block signature documented in `analyze-scrollback.mjs` baseline (40-41 line runs observed).
- Window: per-span (between AC-2 markers OR start/end of buffer) — no global regex over arbitrary length; max scan budget bounded by max scrollback size (1 MiB + 1 MiB rotated = 2 MiB worst-case).

**Acceptance:**
- (U) Given `bytes()` returns the raw on-disk content unchanged, when both `bytes()` and `bytesForReplay()` are called against the same fixture, then `bytes()` output equals disk-bytes byte-for-byte; `bytesForReplay()` may differ.
- (U) Given a fixture buffer with 5×PowerShell-banner-burst within one shell-span (synthetic), when `bytesForReplay()` reads it, then output contains exactly 1 banner + 1 collapse marker `\x1b[2m── 4 earlier banners collapsed ──\x1b[m`.
- (U) Given a fixture buffer with the literal string `PowerShell 7.6.1` mid-stream (echo'd by user inside Claude TUI; NOT preceded by ≥10 CRLFs), when `bytesForReplay()` reads it, then the literal string is preserved (NOT collapsed).
- (U) Given a fixture with 3 banner-bursts, an AC-2 marker, then 3 more banner-bursts, when `bytesForReplay()` reads it, then EACH side of the marker collapses independently (2 collapse-markers + 2 banners total + 1 AC-2 marker preserved). Cross-marker boundary NEVER collapsed.
- (U) Given a malformed OSC sequence with >256 bytes-without-BEL, when `bytesForReplay()` reads it, then the bounded regex fails to match the OSC and the banner is preserved as-is (graceful — no false collapse, no ReDoS).
- (E) Spec 83 AC-3: setup live task, kill→respawn 5 times, navigate away+back → expect one banner + one collapse marker visible in `term.buffer.active`.

### AC-4 (Stage 3) — Footer banner: "Scrollback enthält N beendete Shell-Sessions"

**Goal:** When `EmbeddedTerminal` receives `replay_end` AND the xterm buffer contains ≥2 of the AC-2 marker strings, render a dim-grey footer banner below the terminal surface:
```
Scrollback enthält 3 beendete Shell-Sessions. [Clear history]
```
The "Clear history" button reuses the existing `…` overflow-menu's `Clear terminal history` confirm-modal at [TaskDetailHeader.tsx:400+](../../../client/src/components/external/TaskDetailHeader.tsx#L400) (no new server endpoint, no new modal — `POST /api/terminal/:taskId/clear-scrollback`).

**Per external plan review (2026-05-08) — counting strategy:**
- **Count via `term.buffer.active`, NOT chunk-substring.** WebSocket `data` envelopes can split a marker string across chunks (`\x1b[2m────` in frame 1, `─ shell stopped...` in frame 2 ⇒ a per-chunk substring count would miss it). Instead: at `replay_end`, scan the live xterm buffer (`term.buffer.active`) for the literal `──── shell stopped at` substring. This works because xterm's parser has reassembled the chunks into a contiguous buffer by then.
- **Wording precision:** "beendete" (precise: ENDED sessions, where this marker is) vs the original "frühere" (vague). Maps 1-to-1 to AC-2 marker count.
- **Clear-history reset:** after `clear-scrollback` succeeds, the existing `confirm-modal` already triggers a WS reconnect cycle ⇒ replay_end fires with empty buffer ⇒ count is 0 ⇒ footer hides naturally. Add E2E assertion to confirm.

**Acceptance:**
- (U) Component test: with 3 stop-marker strings present in xterm buffer → footer banner visible with text "Scrollback enthält 3 beendete Shell-Sessions" + clear-history button.
- (U) Component test: with 0 or 1 marker → footer banner NOT visible.
- (U) Component test: after `clear-scrollback` succeeds → footer disappears without component remount.
- (E) Spec 83 AC-4: induce 3 kill→respawn cycles → re-attach → footer shows N=3 → click "Clear history" → confirm-modal opens.

## Out of Scope

- **Disk-scrollback pruning / compaction.** ADR-076 explicitly rejected destructive operations on persisted user-visible data. AC-3's collapse is replay-stream-only; the disk file remains intact.

- **Cycle-rate root-cause investigation.** The 540 banners/13.5h is dominated by `tsx watch` SIGTERM during dev. Production users will see 1-2 banner cycles per day from idle-ceiling. Not blocked on fix.
- **Bilingual disclosure copy.** German-only banner copy is acceptable for now; bilingual deferred.
- **AC-2 marker for `pty.onExit`** (shell crashed naturally). Out of scope — `cleanup()` is hit but `kill()` is not; we only write the marker on the active-kill paths. Future iterate could add it via `cleanup()` if requested.

## Design Notes

- **AC-2 marker visual:** dim-grey ANSI (`\x1b[2m`) + horizontal box-drawing chars `────` (U+2500). If dim-grey doesn't render well in user's xterm theme, fallback is `\x1b[90m` (dark-grey foreground). Both are universal.
- **AC-4 footer:** Tailwind classes — `text-xs text-zinc-500 px-3 py-1 border-t border-zinc-800 bg-zinc-900/50`. Banner sits BELOW the xterm canvas, ABOVE any existing disclosure footer.
- **No new mockups required** — both AC-2 and AC-4 are minor textual additions to existing surfaces.

  5. (AC-2) Closing-flag dedupe: duplicate kill() emits exactly 1 marker (unit test).

  11. (AC-4) Chunk-split marker handling: full-string accumulator captures marker count regardless of WS frame size (replaces fragile tail-overlap logic per gemini review).
  12. (AC-4) Live-data wipe survival: footer count derived from accumulator (NOT `term.buffer.active` post-replay) so pty2's `\x1b[2J\x1b[H` doesn't zero the count (Spec 83 AC-3+AC-4 against live stack).

- **Edge cases NOT probed + why acceptable:**
  1. **Concurrent kill from multiple WS clients on same task** (race within `kill(taskId)`): single-process Node.js — kill() is synchronous after entry.closing=true; second caller sees the deleted entry and short-circuits. Not load-tested under multi-client storm; acceptable because each task has one writer at a time (Plan-D'' contract).
  2. **`pty.onExit` fails to fire** on Windows ConPTY pathological exits (e.g., kill -9 from outside): no marker. Acceptable — that's the same behavior as `pty.kill()` with closing=false (natural exit). Pty is gone; the user sees no banner accumulation either way.
  3. **AC-3 collapse with non-PowerShell shell** (cmd.exe / bash): the bounded regex anchors on `PowerShell N.N.N` literal, so cmd / bash banner-bursts (different shape) won't collapse. Documented as Windows-PowerShell-specific in spec; bash-on-Windows users won't see the symptom (cmd has no version-banner-on-spawn pattern observed; bash doesn't either).
  4. **AC-4 footer with very many markers** (e.g. 50+): no upper bound enforced. Acceptable — `parseInt` of count number is correct + count is one source-of-truth derived from bounded scrollback (max 2 MiB).

- **Confidence-pattern check:** No "are you confident?"-style yes-then-finding pattern fired in this run. All decisions are evidence-driven (external review feedback applied; empirical Spec 83 against live stack).
