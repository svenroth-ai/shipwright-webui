## Lessons-from-v0.8.5 to apply this iterate

1. **E2E "always" at medium = author AND run.** Each stage's empirical Playwright spec is written BEFORE the fix (red), runs against the live stack to prove the bug, then the fix lands and the same spec turns green. No more "manual UAT" justification when a startable surface exists.
2. **Investigation-led ACs need a reproduction test BEFORE a fix.** AC-2 (banner count after revisit) and AC-3 (read-only after revisit) both exhibit on the same trigger (task → board → task) — likely sharing a root cause; same Playwright fixture covers both.
3. **No mechanical safeguards on this project.** `verify_iterate_finalization.py` + Phase-Quality-Stop hook + `surface_verification.py` are all behind uv-based scripts that don't always have PATH; the project gitignores `.shipwright/agent_docs/`. The only gate is discipline.

## Goal

Make the embedded terminal stable across task → board → task navigation: no banner accumulation, no spurious read-only state, no rounded-corner artefacts, no obsolete CTA on the TaskCard. Each AC has a Playwright regression case that proves the bug pre-fix and the contract post-fix. Ships as **v0.8.6**.

## Acceptance Criteria

### AC-1 (Stage 0) — Drop rounded corners on terminal wrapper

User-reported (post-v0.8.5 UAT): "Ecken sind abgerundet. Aber sieht doof aus. Nicht abrunden." Visually the rounded corners are confirmed (Spec 81 AC-1 asserted `borderRadius === "6px"` and passed) but they read as out of place against the rest of the WebUI's mostly-square surface chrome.

**Goal:** Embedded terminal wrapper has NO border radius. The dark padded canvas is square corners.

**Implementation:** drop `rounded-md` from the EmbeddedTerminal outer `<div>`. Drop `rounded-t-md` from the conditional banners (read-only / replay-only / preview-command). Update Spec 81's AC-1 assertion: `borderRadius === "0px"`.

**Files:** `client/src/components/terminal/EmbeddedTerminal.tsx`, `client/e2e/flows/81-v0.8.5-terminal-fixes-smoke.spec.ts` (assertion flip).

**Severity: LOW.** **Estimate: 0.1d.**

### AC-2 (Stage 1) — No banner accumulation on tab revisit

User-reported: "Dann ist der Fehler nun schlimmer mit den 100 Claude Code Headers nach Rückkehr zum Terminal." After v0.8.5 AC-3 (defensive `term.clear()` on `replay_start`), the user expected the 100-banner observation to NOT GROW across revisits. Instead it got worse — implies either:
  - (a) Disk-scrollback is being appended to during the user's active session faster than expected, so each revisit sees a longer history.
  - (b) `term.clear()` doesn't fire reliably (timing race with replay).
  - (c) There's a SECOND replay path I haven't accounted for (re-attach not via `replay_start` but via something else).
  - (d) TaskDetailPage isn't actually unmounting on /board → /tasks/:id, and the existing xterm accumulates without my AC-3 ever firing.

**Investigation steps (Stage 1 BEFORE writing fix code):**

1. Author **Spec 82 AC-2 reproduction**: launch a new-plain task → wait for ready → click into a banner-heavy fixture (or accept the natural Claude-startup banner) → count `Claude Code v2.1.132` substring occurrences in the rendered xterm (via `evaluate(() => document.querySelector(".xterm-rows").textContent)` or similar) → navigate to `/` → wait → navigate back to `/tasks/:id` → wait for ready → count again. **Assert count is ≤ initial+0** (idempotent across navigation).
2. Run Spec 82 AC-2 against the live v0.8.5 stack — should FAIL (bug reproduces).
3. Diagnose root cause:
   - Verify whether TaskDetailPage actually unmounts on /board navigation (mount counter spy in dev mode).
   - Verify whether `term.clear()` runs (data-attribute counter on the wrapper).
   - Verify what envelope sequence the second WS upgrade emits (`replay_start`? Or just live data?).
4. Fix matching the diagnosis. Most likely candidates:
   - **Server:** don't replay scrollback when an attach reuses a still-warm pty (only replay on cold-attach). Track per-pty "last-attach-served-replay" flag.
   - **Client:** explicit `term.reset()` (full RIS) on every WS reconnect, not just `replay_start`. Belt-and-braces if the server emits scrollback via different paths.
   - **Disk:** if (a) is the cause — find the writer that's appending duplicate banners and stop it.
5. Re-run Spec 82 AC-2 — must turn green.

**Acceptance:** After 3 round-trips TaskBoard → Task → TaskBoard → Task, the rendered xterm shows the SAME content as after a single navigation. No accumulation, no missing content.

**Severity: HIGH** (UX confidence breaker; was supposed to be fixed in v0.8.5). **Estimate: 0.5–1d.**

### AC-3 (Stage 2) — No spurious read-only banner on tab revisit

User-reported: "Neu ist noch der read only fehler da. Den haben wir schonmal gefixt." Reproduction recipe: launch task → leave to TaskBoard → return to task → embedded-terminal-readonly banner shows.

The banner renders when `socket.role === "reader"`. The historical fix (`writer-promoted` envelope) handles the StrictMode double-mount race where two WS connections briefly co-exist. The regression suggests:
  - Old WS connection isn't fully released on TaskDetailPage unmount.
  - New WS connection on revisit gets reader role because the server still sees the old conn-token as writer.
  - `writer-promoted` envelope doesn't fire because the old conn closes AFTER the new one has already taken reader.

**Investigation steps (Stage 2 BEFORE writing fix code):**

1. Author **Spec 82 AC-3 reproduction**: same fixture as AC-2 → after the second navigation, assert `getByTestId("embedded-terminal-readonly")` has count 0.
2. Run Spec 82 AC-3 → should FAIL.
3. Diagnose:
   - Server-side: when does the OLD conn-token actually get released? Look at WS close handler in `terminal/routes.ts`. If async cleanup, the new attach may race in.
   - Client-side: does TaskDetailPage's unmount fire `useTerminalSocket`'s cleanup deterministically? React 19 StrictMode quirks?
4. Fix candidates:
   - **Server:** synchronous writer-release in WS `onClose`; OR a "claim" handshake on attach that waits for any pending close.
   - **Client:** `useTerminalSocket` cleanup must close the WS BEFORE the new socket connects (sequential, not concurrent).
   - **Both:** `writer-promoted` envelope on RECONNECT-with-same-task-from-same-tab, not just on multi-tab.
5. Re-run Spec 82 AC-3 → must turn green.

**Likely shared root with AC-2.** If TaskDetailPage doesn't fully unmount + WS doesn't close + new attach gets reader, the symptom set matches both: replay-via-new-mode + reader-role.

**Severity: MID.** **Estimate: 0.5d (could compress if AC-2 root cause covers it).**

### AC-4 (Stage 3) — Remove Terminal button from TaskCard

User-reported: "Terminal Button auf der Card macht auch keinen Sinn mehr. Nur Launch macht Sinn oder Resume, aber Terminal nicht. Klick auf die Karte führt zum Terminal Tab."

v0.8.5 AC-6 removed the Terminal CTA from TaskDetailHeader but missed the TaskCard surface on TaskBoard. Same logic applies: Click on the card itself routes to the task detail page (which lands on Terminal tab if that's the persisted preference); having a separate "Terminal" button is redundant.

**Goal:** TaskCard CTA matrix shrinks to:

| Card-task state | Card CTA |
|---|---|
| `draft` | **Launch** (auto-execute via `dispatchAutoLaunch` after navigation) |
| `awaiting_external_start` / `active` / `done` / `launch_failed` | **(none)** — card body is the click target |
| `idle` | **Resume** (auto-execute --resume) |

**Implementation:** locate the Terminal button in `client/src/components/external/TaskCard.tsx` — remove its branch + the click handler that dispatches `webui:focus-terminal-tab` (which we already removed from TaskDetailPage in v0.8.5 AC-6, so this listener has been dead since v0.8.5).

**Files:** `client/src/components/external/TaskCard.tsx` + tests touching its CTA matrix.

**E2E coverage:** Spec 82 AC-4 — render TaskBoard with a task in `awaiting_external_start`, assert no `cta-terminal` button on the card.

**Severity: MID.** **Estimate: 0.25d.**
