## Acceptance Criteria

### Stage 0 ‚Äî Spec 74 modal flake (NEW, found during v0.8.1 live smoke)

- [ ] **AC-1 (Spec 74 clear-history-modal flake):** Three Spec 74 tests (`Cancel dismisses without action` / `Confirm posts /clear-scrollback + closes modal` / `surfaces 5xx error inline`) currently 30s-timeout because the menu's `e.preventDefault()` keeps the dropdown open while the modal renders, and Playwright's `toBeVisible` on the dialog races the menu's still-active state on Windows ConPTY. **Verified pre-existing on main during v0.8.1 sweep ‚Äî not an iterate regression.** Fix: either close the dropdown explicitly on `clear-history` select (drop the preventDefault, accept the default close behavior, then open modal in a `setTimeout`/`requestAnimationFrame` to escape the menu's dismiss frame), OR add an explicit `data-state="closed"` wait in the test before the modal `toBeVisible` check. Decision documented in ADR. **Files:** `client/src/components/external/TaskDetailHeader.tsx` (~lines 734‚Äì747), `client/e2e/flows/74-auto-launch-disk-persistence.spec.ts` (3 tests).

### Stage 1 ‚Äî AC-2 black-on-black input (carried over from iterate-2026-05-05-post-v0.8-stabilization)

- [ ] **AC-2 (black-on-black input rendering eliminated):** Diagnosis-first via Playwright WS-frame capture against a live Claude Code TUI session inside the embedded terminal: identify the exact escape sequence Claude Code emits for its TUI input-box that collides with the xterm light theme. Diagnosis test lives at `client/e2e/diagnostic/ac2-claude-tui-bytes.diag.ts` (gitignored or `*.diag.ts` suffix; removed pre-merge). Then either (a) extend the xterm theme to remap the offending ANSI slot pair to a brand-warm value, OR (b) detect Claude-Code-TUI deterministically (server-provided session flag, NOT prompt-content sniffing) and switch xterm to a dark theme (`bg=#1a1a1a fg=#f5f0eb`) for the duration of the session ‚Äî applied ONCE at session start, NOT mid-stream. Decision document captures the diagnosis output + chosen approach in ADR. New unit tests: (1) WCAG AA contrast assertion across slot pairs (no fg/bg pair < 4.5:1) AND (2) fixture-driven assertion against the actual diagnosed escape sequence (per OpenAI #6 review finding from v0.8.1 ‚Äî primary regression guard). **Severity: HIGH** ‚Äî typed input is illegible under Claude Code TUI. **Estimate: 0.5‚Äì1d** (diagnosis-first). **Files:** `client/src/components/terminal/EmbeddedTerminal.tsx` theme block (~lines 285‚Äì319), `client/src/components/terminal/EmbeddedTerminal.test.tsx`, `client/e2e/diagnostic/ac2-claude-tui-bytes.diag.ts` (NEW, local-only).

### Stage 2 ‚Äî Image-paste cluster (Tier 1)

- [ ] **AC-3 (Ctrl+V image-paste restored to parity with Alt+V):** Diagnosis-first via WS-frame capture comparing Ctrl+V vs. Alt+V paste paths. Hypothesis: xterm's own keybinding swallows Ctrl+V before the DOM capture-phase listener fires, OR focus-on-canvas vs. focus-on-container affects DOM event delivery. Fix likely involves either disabling xterm's Ctrl+V keybinding (`term.attachCustomKeyEventHandler` returning `false` for Ctrl+V to defer to DOM) or moving the paste handler to `document` level. Test: new unit fixture in `EmbeddedTerminal.test.tsx` driving a synthesized Ctrl+V paste-event with image clipboard payload + asserting the DOM handler runs. **Severity: MED**. **Estimate: 0.5d**. **Files:** `client/src/components/terminal/EmbeddedTerminal.tsx` paste handler (~lines 397‚Äì465).

- [ ] **AC-4 (Image-paste latency reduced to <1s):** Diagnose the 5s Alt+V paste delay via Browser DevTools Network panel + server-side timing logs. Hypotheses: (a) magic-byte mime-sniff stalling on large images, (b) multipart upload roundtrip slow on Windows file-write, (c) fetch-retry loop. Add structured timing logs around `savePastedImage` covering (i) clipboardItem‚ÜíBlob conversion, (ii) HTTP roundtrip, (iii) magic-byte sniff, (iv) fs.writeFile, (v) pty.write of the path. Profile, fix the bottleneck. New e2e spec asserts <1500ms wall-clock from clipboard-event to pty-write. **Severity: MED**. **Estimate: 0.5d**. **Files:** `client/src/components/terminal/EmbeddedTerminal.tsx`, `server/src/terminal/image-paste.ts`, `server/src/terminal/routes.ts`.

- [ ] **AC-5 (Awaiting-launch state lag ‚Äî diagnosis only, fix conditional):** Add a structured diag log in `core/session-watcher.ts` that records each polled JSONL path + match-result + elapsed-ms-since-launch. Reproduce a slash-command launch (`/shipwright-iterate <prompt>`) and time the awaiting‚Üíactive transition. **Decision tree:** if <30s, document expected-behavior in `known_issues.md` (claude needs 5‚Äì15s for first JSONL write + 2‚Äì5s poll cadence = up to 20s total). If >30s, diagnose encoded-cwd mismatch between `task.cwd` and the pty's effective post-`Set-Location` cwd. Pure Claude (`new-plain`) explicitly excluded ‚Äî JSONL-on-first-message is correct behavior. **Severity: MED (conditional)**. **Estimate: 0.5‚Äì1d**. **Files:** `server/src/core/session-watcher.ts`, possibly `server/src/core/launcher.ts`.

## Lessons from v0.8.1 to apply

1. **Live-smoke per stage is mandatory** (`feedback_e2e_specs_run_not_just_write.md`). Each stage's commit gets its own `npm run dev` + relevant e2e spec run. Bug found ‚Üí fix before next stage.
2. **Diagnose-first ACs need an explicit Playwright diagnostic spec** (`client/e2e/diagnostic/*.diag.ts`) that runs before the fix. AC-2 + AC-5 both fall into this category.
3. **AC-12-style "Manual UAT (executed by user)" wording is misleading.** AC-11 here is split: agent runs live-smoke for the new e2e specs (mandatory); user runs the human UAT walk-through (still recommended).
4. **External-review cascade continues:** plan review ‚Üí code review ‚Üí live smoke ‚Üí Codex (if budget). Each stage finds a different bug class. v0.8.1's code-review caught the per-conn watchdog capability bug that the plan-review missed.
5. **Pre-existing test flakes are documented + fixed in their own stage** (AC-1 here = Spec 74). Don't roll regressions of pre-existing bugs into other ACs' commits.
